# Connection: DIY Software Mindset × Model-Code Agent Separation

## Relationship

**Model-Code Agent Separation 是 DIY Software Mindset 在技术实现层面的具体架构。**

- **DIY Software Mindset**: 价值观——拼凑优先，能用就行
- **Model-Code Agent Separation**: 实施架构——明确分工，避免过度工程

两者结合，确保 DIY 模式在技术上可行且高效。

---

## The Anti-Pattern: Over-Engineering

### 违背 DIY 精神的实现

```
如果让 Code Agent 做所有事：

用户："我想记个账"
    ↓
Code Agent：
- 设计数据库 schema
- 设计 API 接口
- 实现用户认证系统
- 实现数据校验
- 实现报表生成
- ...

结果：
- 3 天后才交付
- 复杂的系统架构
- 实际上用户只需要 Excel 级别的功能
```

### 违背能力边界的设计

```
如果让模型做数值计算：

用户："计算本月支出"
    ↓
模型：直接计算
- 浮点数精度问题：¥1,250.999999
- 汇率换算错误
- 统计数据偏差

结果：
- 数据不可信
- 用户失去信任
```

## The DIY-Aligned Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    DIY 原则                                 │
│              简单能用，拼凑优先，安全底线                      │
└─────────────────────────┬───────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                   Model（模型）                              │
│              负责：理解 + 交互 + 文本处理                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  处理用户输入：                                               │
│  "帮我看看这个月花了多少钱在吃饭上"                            │
│      ↓                                                      │
│  理解：查询本月餐饮类别支出                                    │
│      ↓                                                      │
│  决定：需要数值计算 → 调用 Code Agent                         │
│      ↓                                                      │
│  接收结果：{ total: 1250.50 }                               │
│      ↓                                                      │
│  生成回复："本月餐饮花了 ¥1,250.50，                         │
│           比上个月少 ¥200，继续保持！"                        │
│                                                             │
│  💡 DIY 特点：                                               │
│  - 不需要精确理解，大概对就行                                  │
│  - 回复可以口语化、个性化                                     │
│  - 文本查找模糊匹配可接受                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                          │
                          │ 简单调用
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                Code Agent（轻量）                            │
│              负责：数值计算 + 简单数据操作                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  只做最简单的事：                                             │
│  - 从 JSON 文件读取记录                                        │
│  - 筛选条件（日期、类别）                                      │
│  - 精确计算总和                                               │
│  - 返回结果                                                   │
│                                                             │
│  💡 DIY 特点：                                               │
│  - 不需要数据库，JSON 文件就行                                 │
│  - 不需要 ORM，直接操作文件                                    │
│  - 不需要复杂架构，脚本级别                                    │
│  - 但必须保证数值准确                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Examples in DIY Context

### 场景 1: 个人记账

```
DIY 级别实现：

数据存储：
- 不用数据库
- 用 JSON 文件：data/expenses.json

Code Agent 职责：
- 读取 JSON
- 过滤日期和类别
- sum() 计算
- 写回 JSON

模型职责：
- 理解"吃饭"=餐饮类别
- 理解"这个月"=当前月份
- 生成鼓励/提醒的回复
- 处理模糊查询（"那个咖啡的钱"）
```

### 场景 2: 待办清单

```
DIY 级别实现：

数据存储：
- 纯文本文件，每行一个任务

Code Agent 职责：
- 读取文件
- 添加/删除行
- 简单的正则匹配

模型职责：
- 理解优先级（"重要的"）
- 理解时间（"下周的"）
- 生成分类建议
- 模糊搜索任务
```

### 场景 3: 简单的数据分析

```
用户："帮我算一下这个月每天平均花多少"

模型：
- 理解需求
- 决定需要：总支出 / 天数

Code Agent：
- 读取记录
- 精确计算：total / days
- 返回：¥41.67

模型：
- "平均每天 ¥41.67"
- "工作日平均 ¥35，周末平均 ¥80"
- "建议：控制周末消费"
```

## Key Principles

### 1. 复杂度控制

```
传统：系统复杂度随功能线性增长
DIY：复杂度保持恒定（模型复杂度 + 简单脚本）
```

### 2. 数据存储简化

```
传统：数据库 + ORM + 迁移脚本
DIY：JSON 文件 / CSV / 纯文本

传统：事务、索引、备份
DIY：定期复制一份文件就行
```

### 3. 功能扩展方式

```
传统：修改架构、添加模块、更新接口
DIY：模型理解新意图 → 调用新的简单脚本

不需要重构，只需要添加新的计算函数
```

---

## Success Criteria

### DIY 架构的检验标准

- ✅ 新功能可以在 10 分钟内添加
- ✅ 不需要修改"架构"
- ✅ 数据存储是简单的文件
- ✅ 数值计算是准确的
- ✅ 交互是流畅的（模型负责）
- ✅ 没有安全漏洞
