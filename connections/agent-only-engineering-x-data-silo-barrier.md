# Connection: Agent-Only Engineering × Data Silo Barrier

## Relationship

**Data Silo Barrier 是 Agent-Only Engineering 场景下的核心约束。**

在 Agent-Only 开发中：
- Agent 可以判断**技术可行性**（通过 WebSearch）
- 但 Agent 无法判断**数据可达性**（是否被锁在孤岛中）
- 这导致 Mock 可能基于"技术可行"的假设，忽视了"数据不可达"的现实

---

## The Problem

### 传统开发

```
工程师："你想获取微信聊天记录？不可能，微信不开放 API"
    ↓
早期阻止不切实际的需求
```

### Agent-Only 开发（无干预）

```
Agent (WebSearch): "Python 可以读取本地数据库文件..."
    ↓
技术判断：可行 ✅
    ↓
NL Mock: 模拟"微信数据分析功能"
    ↓
Demo 看起来很完美
    ↓
用户："对，这就是我想要的！"
    ↓
DDDW 启动
    ↓
实际：微信数据加密，无法读取
    ↓
工程失败（但 Agent 无法提前知道）
```

---

## The Solution: Interactive Risk Exposure

将数据孤岛问题从"后台判断"转为"前台交互"

```
用户："我想分析我的微信聊天数据"
    ↓
Agent: "这个功能涉及获取微信数据，我需要确认几个前提"
    ↓
交互式 Demo 展示：

┌─────────────────────────────────────────┐
│  微信数据分析功能 - 前提条件确认          │
├─────────────────────────────────────────┤
│                                         │
│  📊 功能预览（基于模拟数据）              │
│  ┌─────────────────────────────────┐   │
│  │ 聊天频次分析                     │   │
│  │ 情感趋势图                       │   │
│  │ 关键词云                        │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ⚠️ 数据获取前提（需要您确认）           │
│  ┌─────────────────────────────────┐   │
│  │                                 │   │
│  │ 1. 微信是否提供数据导出功能？    │   │
│  │    [帮我查一下]                  │   │
│  │                                 │   │
│  │    Agent WebSearch 结果：        │   │
│  │    "微信仅支持聊天记录迁移，      │   │
│  │     不提供结构化数据导出"        │   │
│  │                                 │   │
│  │ 2. 您是否能接受替代方案？        │   │
│  │    ○ 手动复制粘贴关键对话        │   │
│  │    ○ 仅分析您主动记录的内容      │   │
│  │    ○ 放弃此功能                  │   │
│  │                                 │   │
│  └─────────────────────────────────┘   │
│                                         │
└─────────────────────────────────────────┘
```

**关键转变**：
- Agent 不假装知道用户能获取什么数据
- 而是引导用户在体验中面对数据孤岛的现实
- 让用户和 Agent 一起寻找可行的替代方案

## Data Availability Assessment Flow

```
用户提出涉及外部数据的需求
    ↓
Agent 识别数据依赖
    ↓
┌─────────────────────────────────────────┐
│ Step 1: 技术可行性                      │
│ (Agent 通过 WebSearch 判断)              │
│ "抓取网页数据技术上可行"                 │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│ Step 2: 数据可达性评估                   │
│ (Agent 有限的判断能力)                   │
├─────────────────────────────────────────┤
│                                         │
│ 可搜索确认：                             │
│ - API 是否开放？                         │
│ - 是否有官方数据导出？                    │
│ - 反爬策略严格程度？                      │
│                                         │
│ 无法确认（必须询问用户）：                │
│ - 用户是否有 API 权限？                   │
│ - 用户是否有商业授权？                    │
│ - 用户是否愿意手动提供数据？              │
│                                         │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│ Step 3: 交互式确认                       │
│ (在 Demo 中暴露风险)                     │
│                                         │
│ 让用户明确选择：                         │
│ - 我能提供数据的方式是什么？              │
│ - 如果不能，我能接受什么降级方案？         │
│                                         │
└──────────────────┬──────────────────────┘
                   │
                   ▼
              确认的数据方案
              成为需求文档的一部分
```

## Integration with Existing Concepts

### 与 [[mock-feasibility-boundary]] 的结合

传统可行性边界：
- 技术可行性
- 成本可行性
- 时间可行性

Agent-Only 场景新增：
- **数据可达性** ← Data Silo Barrier
- **权限确认** ← 必须用户自我声明

### 与 [[demo-as-requirements]] 的结合

Demo 不仅是功能验证，更是**数据可行性验证**

```
Demo 交付物包括：
├── 功能演示
├── 用户确认的交互流程
├── ⚠️ 数据获取方案（用户已选择）
└── ⚠️ 降级方案（如果数据不可得）
```

---

## Key Insight

> **在 Agent-Only 开发中，数据孤岛问题不能仅由 Agent 判断，必须通过交互让用户参与解决。**

传统工程师：
- 凭经验知道"微信数据拿不到"
- 早期就阻止不切实际的需求

Agent + 交互式设计：
- Agent 不知道（或不确定）
- 但通过 Demo 让用户也意识到"这确实拿不到"
- 共同寻找务实的解决方案

这不是缺陷，而是一种**分布式认知**——Agent 和用户各自贡献所知，共同完成可行性评估。
