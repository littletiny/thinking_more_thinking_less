# Connection: DIY Software Mindset × Hard Constraints System

## Relationship

**DIY Software Mindset 重新校准了 Hard Constraints System 的约束强度。**

在 DIY 模式下：
- **大部分约束都可以放松** —— 性能、可维护性、架构都不重要
- **安全约束必须保留** —— 这是唯一的底线
- **可行性的门槛大幅降低** —— "能用就行"

---

## Constraint Reassessment

### 传统 Hard Constraints vs DIY Hard Constraints

| 约束类型 | 传统评估 | DIY 评估 |
|---------|---------|---------|
| **技术可行性** | 需要稳健方案 | 能跑就行 |
| **性能要求** | 必须达标 | 无所谓 |
| **可扩展性** | 需要考虑 | 不需要 |
| **维护成本** | 需要评估 | 不考虑 |
| **代码质量** | 需要规范 | 无要求 |
| **安全漏洞** | ⚠️ **必须检查** | ⚠️ **必须检查** |
| **数据泄露** | ⚠️ **必须检查** | ⚠️ **必须检查** |

## Implications for Workflow

### 加速的 Exploration Phase

```
传统：
用户意图 → 详细可行性评估 → 架构设计 → Demo
   ↑______________________________|
              （耗时）

DIY：
用户意图 → 安全检查 → 找现成方案 → 拼凑 Demo
   ↑_______________________|
          （快速）
```

### Agent 角色的转变

```
传统 Agent：
- 架构师（设计系统）
- 工程师（编写代码）
- 评审（检查质量）

DIY Agent：
- 采购员（找现成工具）
- 胶水工人（连接组件）
- 安全员（检查漏洞）
```

### 组装优先策略

```
面对用户需求，Agent 的决策树：

1. 安全检查（必须通过）
   ↓
2. 有现成 API？→ 直接调用
   ↓ 否
3. 有开源库？→ 引入使用
   ↓ 否  
4. 能复制粘贴？→ 拿来用
   ↓ 否
5. 必须自己写？→ 写最短的代码
```

## Success Metrics

### DIY 项目的完成标准

```
传统项目 checklist：
□ 架构合理
□ 代码规范
□ 测试覆盖
□ 性能达标
□ 文档完整
□ 安全无漏洞

DIY 项目 checklist：
□ 能跑起来
□ 解决了我当前的问题
□ 没有安全漏洞
□ （其他都是可选）
```

---

## Key Insight

> **DIY 模式不是"低质量"，而是"恰到好处"——用最小的成本解决当前的问题。**

对于一人项目：
- 过度工程是浪费
- 提前优化是毒药
- 简单能用是美德

Agent 的作用：
- 不是"生成完美的代码"
- 而是"快速拼凑可用的工具"
- 同时"守住安全底线"
