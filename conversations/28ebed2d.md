## 🎯 WHY：为什么需要需求澄清？

### 1. 传统工具链的断裂

```
传统流程的问题：

想法 ──────→ 设计/代码 ──────→ 产品
     ↑         ↑              ↑
   口头描述   Figma/v0.dev    上线
```

**断裂点**：
- **想法→设计**：需要设计师翻译，业务方无法直接验证
- **设计→代码**：设计稿不能运行，开发者理解有偏差
- **无决策记录**：整个过程没有记录"为什么这样设计"的历史

### 2. 需求传递中的信息损耗

| 环节 | 问题 | 结果 |
|------|------|------|
| 业务方→产品经理 | 口头描述模糊 | "我想要一个看板" |
| 产品经理→设计师 | 理解偏差 | 设计稿不是业务方想要的 |
| 设计师→开发者 | 交互细节缺失 | 开发成果与预期不符 |
| 开发→上线 | 技术约束未提前暴露 | 返工或妥协 |

### 3. 现有工具的错位

```
现有工具定位：

v0.dev ─────────→ 生成生产代码（开发者导向）
                    ↓
Figma AI ───────→ 生成设计稿（设计师导向）
                    ↓
Coze/n8n ───────→ 工作流自动化（执行导向）

缺失：专门的"需求验证"层（业务人员导向）
```

## ⚙️ HOW：MockCraft 如何解决？

### 核心机制：三明治架构

### 具体解决方式

#### 1. **可丢弃原型（Disposability）**

```
传统方式的问题：
• 设计稿/代码修改成本高 → 用户不敢试错 → 接受"差不多就行"
• 产出物被视为"资产" → 心理负担重 → 难以快速迭代

MockCraft 的解决：
产出物是可丢弃的 → 用户没有心理负担 → 愿意试错
      ↓
修改成本极低（秒级重新生成）→ 鼓励迭代 → 更快找到正确方向
      ↓
不替代生产工具 → 与 Figma/v0 互补而非竞争
```

#### 2. **渐进式澄清（Progressive Clarification）**

```
第一轮：捕捉意图（关键词提取）
   "我要一个看板" → 提取：看板、任务管理

第二轮：生成假设原型（快速验证方向）
   生成 Demo A → 用户反馈："不对，要按区域分组"

第三轮：多场景扩展（覆盖边界情况）
   按区域分组看板 → 模拟空数据/大量数据/错误状态

第四轮：可行性检查（技术约束）
   检查：是否需要实时协作？数据量多大？

第五轮：生成交付物（PRD + Demo）
   输出：确认的需求文档 + 可交互原型
```

#### 3. **过程即文档（Process as Documentation）**

```
传统 PRD：
先写文档 → 再开发 → 文档与实际脱节

MockCraft 的方式：
迭代 1: "我要看板" → 生成 Demo A → 反馈"不对，要按区域"
迭代 2: "按区域分组" → 生成 Demo B → 反馈"加上筛选"
迭代 3: "带筛选的分组看板" → 生成 Demo C → 确认"对！"

自动生成文档：
## 需求演进
- 初始：模糊的"看板"
- Iter1 发现：需要按区域分组（原需求未提及）
- Iter2 发现：需要筛选功能（交互细节）

## 最终确认
功能：分组看板 + 筛选器
数据：模拟 3 个区域、20 条记录
风险：无（无复杂技术约束）
```

#### 4. **与现有工具的区别**

| 维度 | Figma | v0.dev | **MockCraft** |
|------|-------|--------|---------------|
| 目标用户 | 设计师 | 开发者 | **业务人员/PM** |
| 产出物性质 | 设计稿 | 生产代码 | **可丢弃原型** |
| 修改成本 | 中（调设计） | 高（改代码） | **极低（重新生成）** |
| 记录决策历史 | ❌ 无 | ❌ 无 | **✅ 自动生成** |
| 可行性检查 | ❌ 无 | ❌ 运行时暴露 | **✅ 前置检查** |

## ⚠️ 关键设计原则

### 1. **"可丢弃性"降低心理门槛**
- 明确告诉用户：这只是验证工具，不是最终产品
- 修改成本趋近于零，鼓励大胆试错

### 2. **自然语言输入**
- 业务人员无需学习设计工具或编程
- 像聊天一样描述需求

### 3. **实时可视化反馈**
- 每轮迭代都有可交互的 Demo
- 从"想象"变成"看见"

### 4. **自动记录决策历史**
- 澄清过程本身就是文档
- 开发者可以看到"为什么这样设计"的完整脉络

---

## 📌 一句话总结

> **MockCraft 不是让业务人员"设计"或"开发"，而是让他们能够"验证自己的想法是否正确"——在投入昂贵的开发资源之前。**
