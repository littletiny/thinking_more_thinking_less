# MockCraft 设计定位讨论

## 对话主题

从 "mock as demo" 概念出发，探讨如何构建一个基于自然语言的 UI/UX 原型生成系统，最终提炼出"需求澄清层（Requirement Clarification Layer）"的结构性定位。

## 核心问题

1. 如何将模糊需求转化为基于 Web 的 UX 交互？
2. 市面上是否有类似产品？是否存在结构性空位？
3. 该系统的技术可行性、商业模式、开源价值分别如何？

## 关键洞察

### 洞察 1：三明治结构定位

```
上游：意图层（业务人员的模糊想法）
  ↓
中游：需求澄清层（MockCraft）
  - 自然语言输入
  - 生成可交互原型（不可用于生产）
  - 快速迭代验证
  - 记录决策历史
  ↓
下游：实现层（Figma/v0/Coze/n8n/代码）
```

### 洞察 2：市面上存在结构性空位

现有工具：
- v0.dev / Lovable：生成生产代码（开发者导向）
- Figma AI / Uizard：生成设计稿（设计师导向）
- Coze / n8n：工作流自动化（执行导向）

缺失：专门的"需求验证"层（业务人员导向）

### 洞察 3：商业产品的风险 vs 开源项目的价值

**商业产品视角（高风险）**：
- 伪需求风险：用户可能不需要"澄清"，只需要"快速看到成果"
- 产出物价值存疑：Demo 验证后即被丢弃，开发者可能不读迭代历史
- 商业模式不清：按生成次数收费用户嫌贵，免费则成本难覆盖
- 大厂降维打击：Vercel/Figma 增加一个功能即可覆盖

**开源项目视角（高价值）**：
- 方法论落地：让知识库的 mock-driven-validation 理论有可运行实现
- 教育工具：用于教授"需求工程"，演示从模糊到清晰的过程
- 生态补位：填补开源工具链中"需求层"的空白
- 私有化部署：企业内网使用，数据不出境

### 洞察 4：关键设计哲学——可丢弃性

RCL 的核心差异：
- 产出物是"可丢弃的" → 用户无心理负担 → 愿意试错
- 修改成本极低（秒级）→ 鼓励迭代 → 更快找到正确方向
- 不替代生产工具 → 与 Figma/v0 互补而非竞争

## 讨论过程

### 第一阶段：问题定义

用户提问：如何将模糊需求转为 Web UX 交互？
- 回顾知识库中的 mock-as-demo-environment、natural-language-mock 等概念
- 提出多 HTML Demo 架构：单场景 → 跨场景流程 → 版本对比

### 第二阶段：结构定位

探讨系统架构：
- 三明治模型：上游意图层 → 中游澄清层 → 下游实现层
- 类比 Figma：面向非专业人员的"自然语言驱动的 Figma"
- 类比文档："可执行的需求文档"（Executable PRD）

### 第三阶段：竞争分析

市面产品调研：
- v0.dev：生成 React 代码，开发者导向
- Galileo AI / Uizard：生成设计稿，设计师导向
- 结论：存在结构性空位——没有专门的"需求澄清"工具

### 第四阶段：批判性审视

残酷自我质疑：
- 用户真的需要"澄清需求"吗？还是只需要快速看到成果？
- 用户有耐心迭代 3-5 轮吗？
- 产出物（迭代历史文档）真的有人看吗？
- ChatGPT + 截图 + Word 是否已经是免费替代方案？

结论：作为商业产品风险极高，但作为开源方法论工具有独特价值。

### 第五阶段：开源重构

价值转换：
- 从"卖工具"到"建立标准"
- 从"服务大众"到"教育/研究/集成"
- 用户自备 API Key 解决成本问题
- 纯前端/可自托管解决隐私问题
