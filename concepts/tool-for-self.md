# Tool for Self / 工具为己

## One Sentence
工具的第一性原理是解决创造者自己的问题，而非服务假想的市场或用户；开源分享是副产品，不是目的。

---

## Why It Matters

### 产品思维的陷阱

```
常见的思维路径：
"有个想法" → "市场多大？" → "竞品分析" → "商业模式？" → 
"如何获客？" → "如何变现？" → 陷入无尽的商业计划

结果：
• 从未真正开始
• 或者做出了没人用的产品
• 为了解决"假想用户"的问题，耗尽资源
```

### 工具思维的本质

```
正确的思维路径：
"我遇到一个问题" → "现有工具不够好" → 
"我自己做一个" → "解决了我的问题" → 
"顺便开源，如果有人也需要" → 结束

结果：
• 至少解决了 1 个人的真实问题（自己）
• 没有维护压力（自己用，有问题自己修）
• 开源是礼物，不是负担
```

## Core Principles

### 1. 己所不欲，勿施于人（反向）

```
自己都不用的工具，不要指望别人用。

检验标准：
• 如果你自己不会每天用它，那它不值得存在
• 如果你更愿意用现有工具（即使不完美），那没必要做新的
• 只有当现有工具"痛苦到无法忍受"时，才值得动手
```

### 2. 一人用户即成功

```
成功标准：
• 商业产品：DAU > X，收入 > Y，增长 > Z
• 工具为己：我每天都在用，它让我更高效

即使只有 1 个用户（自己），也是成功。
其他用户是意外收获，不是 KPI。
```

### 3. 没有维护压力

```
为自己做工具的优势：
• 不需要客服（有问题自己修）
• 不需要文档（自己知道怎么用）
• 不需要升级（够用就行，不强求新功能）
• 不需要兼容（只支持自己的使用场景）

开源后的心态：
• "我开源了，你可以用，但不保证维护"
• 有人提 PR，高兴就合，不高兴就不合
• 没有道德绑架，没有社区压力
```

### 4. 简单到不能再简单

```
为自己做工具，可以极致精简：
• 不需要多用户（就我一个）
• 不需要权限管理（全部开放）
• 不需要漂亮 UI（够用就行）
• 不需要所有功能（只要我最常用的）

对抗功能蔓延的最佳方式：
"这个功能我自己需要吗？不需要就不做。"
```

## Application: MockCraft Re-framed

### 之前的错误定位

```
产品思维下的 MockCraft：
• 目标：普惠大众，让非技术人员生成工具
• 竞争：与 n8n/Coze/飞书比较
• 商业模式：SaaS 或开源捐赠
• 成功标准：用户数、Star 数

结果：
陷入无尽的"市场分析"，忘记了自己是否真的需要它。
```

### 正确的工具思维定位

```
工具为己的 MockCraft：
• 目标：解决我自己的记录痛点
• 竞争：与"手动配置飞书"比较
• 商业模式：无，就是自己的工具
• 成功标准：我每天用它吗？

核心问题：
"我真的觉得手动配置飞书很痛苦吗？
 我真的需要一个语音生成配置的工具吗？
 如果答案是 Yes，就做。如果 No，就停。"
```

## Contrast: Product vs Tool

| 维度 | 产品思维 | 工具为己 |
|------|---------|---------|
| **起点** | 市场机会 | 个人痛点 |
| **用户** | 假想的大众 | 自己（第 1 用户） |
| **功能** | 尽可能多（满足各种需求） | 尽可能少（只满足我的需求） |
| **设计** | 追求普适 | 追求个人顺手 |
| **维护** | 团队责任 | 个人爱好 |
| **开源** | 获客/生态手段 | 分享礼物 |
| **成功** | 用户数、收入 | 我每天都在用 |
