# DIY System Blindspots / DIY 系统盲点

## One Sentence
在反软件工程的设计中，可能存在被忽视的盲点：数据迁移、备份、多设备、长期维护等问题尚未解决，需要系统性审视。

---

## Why It Matters

### 反工程设计的盲区

当我们抛弃工程规范时，也可能抛弃了解决某些问题的标准方案。这些被忽视的问题可能在后期爆发。

### 需要回答的问题

这套 DIY 系统是否遗漏了：
1. 数据生命周期管理
2. 多设备场景
3. 长期维护策略
4. 故障恢复机制
5. 隐私合规边界

## Blindspot Analysis

### Blindspot 1: Data Migration & Schema Evolution

**问题**: 
```
Week 1: 数据结构 { name, value }
Week 10: 需要 { name, value, category, timestamp }

JSON 文件如何迁移？
- 手动改文件？（用户会吗？）
- Code Agent 自动迁移？（需要检测版本）
- 直接丢弃旧数据？（可接受吗？）
```

**传统工程方案**: 数据库迁移脚本（migration）

**DIY 现状**: 未定义

**可能方案**:
```javascript
// 在 DIY-001 中增加数据版本
{
  "_version": 2,
  "_migrated": "2024-03-08",
  "items": [...]
}

// 启动时自动迁移
if (data._version === 1) {
  data.items = data.items.map(i => ({...i, category: 'default'}));
  data._version = 2;
}
```

### Blindspot 2: Backup Strategy

**问题**:
```
JSON 文件存在本地
- 电脑坏了怎么办？
- 误删了怎么办？
- 想回滚到昨天怎么办？
```

**传统工程方案**: 自动云备份、版本控制

**DIY 现状**: 未定义

**可能方案**:
```
选项 A: Git 版本控制（对非程序员友好吗？）
选项 B: 自动复制备份文件（data.json.bak.1, .bak.2）
选项 C: 导出功能（让用户手动备份）
选项 D: 接受风险（明确告知用户）
```

---

### Blindspot 3: Multi-Device Scenario

**问题**:
```
用户在手机记账，想在电脑查看

本地 JSON 文件如何同步？
- 手动复制文件？（麻烦）
- 云同步（坚果云/Dropbox）？（依赖外部）
- 自建同步服务器？（太复杂）
- 用 IM Bot 做数据中枢？（有网络延迟）
```

**传统工程方案**: 云数据库 + 实时同步

**DIY 现状**: 未定义

**可能方案**:
```
推荐：IM Bot 作为"无状态"同步器
- 数据存储在聊天记录中
- 任何设备都能通过 Bot 访问
- 符合"IM 优先"原则
```

---

### Blindspot 4: Performance Boundary

**问题**:
```
JSON 文件从 100 条增长到 10,000 条
- 加载变慢
- 内存占用增加
- 全量保存耗时

什么时候必须升级？
- 传统工程：提前设计扩展性
- DIY：遇到性能问题再说？
```

**需要定义**:
```
DIY 性能边界：
- JSON 文件 < 10MB
- 记录数 < 10,000
- 加载时间 < 3秒

超过边界时的策略：
- 归档旧数据（按年月分文件）
- 升级到 SQLite（保留 DIY 简洁）
- 或明确告知"此工具不适合大数据量"
```

---

### Blindspot 5: Error Recovery

**问题**:
```
JSON 文件损坏（写入中断/磁盘故障）
- 无法 parse
- 数据丢失

如何处理？
- 传统工程：事务、WAL、自动恢复
- DIY：？
```

**可能方案**:
```javascript
// 自动备份策略
function safeWrite(file, data) {
  const backup = file + '.bak';
  if (fs.existsSync(file)) {
    fs.copyFileSync(file, backup);  // 写前备份
  }
  fs.writeFileSync(file + '.tmp', JSON.stringify(data));
  fs.renameSync(file + '.tmp', file);  // 原子替换
}
```

### Blindspot 6: Privacy & Compliance

**问题**:
```
即使是个人项目，也可能涉及：
- 记账（财务数据）
- 健康记录（敏感信息）
- 联系人信息

GDPR/个人信息保护法是否适用？
- 存储在 IM 平台（微信/Telegram）的数据归谁？
- 本地数据丢失算谁的责任？
```

**DIY 需要明确**:
```
- 数据存储位置（本地/云端/IM）
- 用户知情同意
- 数据导出/删除方式
- 免责声明（工具性质）
```

---

### Blindspot 7: Long-term Maintenance

**问题**:
```
用户用了半年，积累大量数据
然后发现：
- 脚手架版本更新了
- Node.js 版本不兼容
- 依赖库有安全漏洞

如何维护？
- 传统工程：版本管理、依赖更新
- DIY：用户会自己处理吗？
```

**可能方案**:
```
DIY 维护策略：
1. 脚手架长期稳定（不追新）
2. 少依赖 = 少维护
3. 提供"导出所有数据"功能（随时可迁移）
4. 接受"用完即弃"（数据导出后重新搭建）
```

---

### Blindspot 8: Debugging & Logging

**问题**:
```
出错了怎么办？
- 没有日志系统
- 没有错误上报
- 用户描述不清楚

如何调试？
```

**可能方案**:
```
DIY 调试策略：
- console.log 保留（用户可以提供截图）
- 数据文件可查看（让用户发 JSON 文件）
- 提供"诊断模式"（输出详细日志）
```

---

### Blindspot 9: Exit Strategy

**问题**:
```
用户不想用这个工具了
数据如何导出？

格式：
- JSON（原始）
- CSV（通用）
- Excel（用户友好）
```

**DIY 必须提供**:
```
"导出我的数据"功能
- 生成标准格式文件
- 让用户能迁移到其他工具
- 符合"用户拥有数据"原则
```

---

### Blindspot 10: Scope Creep

**问题**:
```
用户一开始："我只是想记个账"
一个月后："能不能自动同步银行卡？"
三个月后："能不能做财务分析报表？"

DIY 工具如何应对功能膨胀？
- 接受？（违背简单原则）
- 拒绝？（用户体验差）
- 重新做一个？（数据迁移问题）
```

**可能方案**:
```
明确边界：
- "此工具适合简单场景"
- 复杂需求建议用专业软件
- 提供数据导出，方便迁移
```

## Summary: 10 Blindspots

| # | 盲点 | 紧急程度 | 可能方案 |
|---|------|---------|---------|
| 1 | 数据迁移/Schema 演变 | 中 | 版本号+自动迁移 |
| 2 | 备份策略 | 高 | 自动备份文件 |
| 3 | 多设备同步 | 高 | IM Bot 做中枢 |
| 4 | 性能边界 | 中 | 定义边界+归档策略 |
| 5 | 错误恢复 | 高 | 写前备份 |
| 6 | 隐私合规 | 中 | 明确告知+导出功能 |
| 7 | 长期维护 | 低 | 少依赖+数据导出 |
| 8 | 调试/日志 | 低 | console.log+诊断模式 |
| 9 | 退出策略 | 高 | 必须提供数据导出 |
| 10 | 功能膨胀 | 中 | 明确边界+迁移支持 |
