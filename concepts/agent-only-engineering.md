# Agent-Only Engineering / Agent 即工程团队

## One Sentence
在 AI Native 开发流程中，工程可行性评估完全由 Agent 承担——依赖 WebSearch 和内置知识库判断技术可达性，且必须在 Mock Demo 阶段通过交互式设计暴露数据孤岛等隐性约束。

---

## Why It Matters

### 传统 vs Agent-Only 场景

| 维度 | 传统开发 | Agent-Only 开发 |
|------|---------|----------------|
| **可行性评估** | 人类工程团队 | Agent (WebSearch + 知识库) |
| **知识来源** | 经验 + 调研 | 实时搜索 + 训练知识 |
| **数据可达性** | 工程师了解行业生态 | Agent 可能不知晓封闭生态 |
| **风险暴露** | 会议讨论 | 必须在 Demo 中交互式呈现 |

### 特殊挑战

**挑战 1: Agent 的知识盲区**
```
用户："我想抓取抖音用户的公开视频数据"
Agent (WebSearch): "Python 可以爬取网页..."
    ↓
Demo 生成，看似可行
    ↓
实际：抖音反爬严格，API 不开放，法律风险高
    ↓
Agent 没有足够信息判断"数据孤岛"问题
```

**挑战 2: 数据孤岛的隐蔽性**
```
技术可达：✅ 抓取网页数据技术上可行
数据可达：❌ 对方不开放，反爬，或需授权
Agent 判断：可能误判为"可行"
实际：需要用户自己确认数据获取途径
```

**挑战 3: 没有人类工程师兜底**
```
传统：Demo → 工程团队评审 → 发现风险 → 调整
Agent-Only：Demo → 直接进入 DDDW → 风险可能在后期才暴露
```

## Core Mechanism

### Agent 能力边界声明

```
Agent Engineering Capability:
├── ✅ 技术可行性评估
│   └── WebSearch + 知识库可以判断
│       - "这个功能当前技术能否实现"
│       - "实现复杂度和常见方案"
│       
├── ⚠️ 部分可评估
│   └── 需要明确查询
│       - API 是否开放（需搜索文档）
│       - 第三方服务条款（需搜索）
│       
├── ❌ 无法评估
│   └── 必须依赖用户提供
│       - 用户是否有数据访问权限
│       - 特定组织的内部系统接口
│       - 商业合作/授权状态
│       
└── 🎯 关键认知
    └── Agent 的评估是"基于公开信息的最佳判断"
        不是"保证可行"
```

### 交互式风险暴露设计

**原则**: 在 Demo 中让用户"体验"到限制，而非仅仅"告知"

#### Pattern 1: 假设声明式交互

```
Demo 界面展示：
┌─────────────────────────────────────────┐
│  功能：自动同步您的淘宝订单             │
├─────────────────────────────────────────┤
│                                         │
│  📋 订单列表（模拟数据）                 │
│  ┌─────────────────────────────────┐   │
│  │ 订单号: TB123456                 │   │
│  │ 商品: iPhone 15                  │   │
│  │ 金额: ¥5999                      │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ⚠️ 重要假设（需要您确认）              │
│  ┌─────────────────────────────────┐   │
│  │ □ 您有淘宝开放平台的 API 权限    │   │
│  │ □ 或您能导出订单 CSV 文件        │   │
│  │ □ 或您同意使用浏览器插件方式抓取 │   │
│  └─────────────────────────────────┘   │
│                                         │
│  [以上选项都不符合 - 讨论替代方案]       │
│                                         │
└─────────────────────────────────────────┘
```

**效果**: 用户必须面对"数据从哪里来"的问题

#### Pattern 2: 风险情景模拟

```
Demo 展示"最乐观情况"和"现实约束情况"：

┌────────────────────────────────────────────┐
│  场景对比                                   │
├────────────────┬───────────────────────────┤
│  理想情况      │  现实约束                  │
├────────────────┼───────────────────────────┤
│  ✅ 自动同步   │  ⚠️ 需每月手动导出 CSV    │
│  ✅ 实时更新   │  ⚠️ 延迟 1-24 小时        │
│  ✅ 完整数据   │  ⚠️ 部分字段可能缺失      │
└────────────────┴───────────────────────────┘

您能接受哪种方案？
[接受现实约束] [需要理想方案 - 讨论可行性] [放弃此功能]
```

#### Pattern 3: 数据源确认流程

```
对于涉及外部数据的功能，强制流程：

用户："我想要分析我的 Spotify 听歌数据"
    ↓
Agent: "这个功能涉及获取您的 Spotify 数据"
    ↓
展示交互式清单：

┌─────────────────────────────────────────┐
│  数据源确认清单                          │
├─────────────────────────────────────────┤
│                                         │
│  1. Spotify 是否提供数据导出？          │
│     [帮我查一下] ← 点击后 Agent WebSearch│
│                                         │
│     搜索结果：                           │
│     "Spotify 提供'隐私数据下载'功能，    │
│      可在 Account > Privacy 中申请，     │
│      处理时间 1-30 天"                  │
│                                         │
│  2. 您是否愿意等待数据导出？            │
│     [愿意] [不愿意 - 看替代方案]         │
│                                         │
│  3. 或是否接受简化版（仅当前播放）？    │
│     [接受简化版] [坚持完整数据]          │
│                                         │
└─────────────────────────────────────────┘
```

#### Pattern 4: "已知未知"标注

```yaml
# Demo 元数据（用户可见）
demo_capabilities:
  fully_supported:
    - "本地数据处理"
    - "用户主动输入"
    
  with_assumptions:
    - feature: "第三方数据同步"
      assumption: "用户能提供数据访问方式"
      uncertainty: "需要用户确认具体途径"
      
  known_unknowns:
    - question: "目标平台是否开放 API"
      agent_capability: "可通过 WebSearch 查询"
      user_action: "确认商业授权状态"
      
  unknown_unknowns:
    - disclaimer: "以下情况可能导致功能不可用："
      items:
        - "目标平台突然关闭接口"
        - "反爬策略升级"
        - "用户账号权限变更"
```

## Integration with Workflow

### 修改后的 Exploration Phase

```
用户意图
    ↓
Agent 初步评估（WebSearch + 知识库）
    ↓
技术可行性：✅ / ❌
    ↓
如果 ✅ → 识别数据依赖
    ↓
设计 Demo 时，嵌入"风险暴露交互"
    ↓
用户体验 Demo
    ↓
用户被迫面对数据/授权问题
    ↓
用户选择：
  - 接受约束继续
  - 提供替代数据源
  - 放弃或修改需求
    ↓
迭代直到双方对齐
    ↓
交付物包括：
  - Demo
  - 需求文档
  - 风险假设清单（用户已确认）
  - 数据获取方案（用户已选择）
```

### DDDW 启动条件（Agent-Only 场景）

新增必要检查项：

```
✅ Demo 通过用户验证
✅ 需求文档完整
✅ 用户明确确认
✅ ⚠️ 关键风险假设已标注  ← 新增
✅ ⚠️ 数据获取方案已确认  ← 新增
✅ ⚠️ 用户理解并接受"已知未知" ← 新增
```

## Key Principles

### 1. 透明化 Agent 能力边界

不要假装 Agent 知道一切：

```
错误："这个功能肯定能实现"
正确："基于我的搜索，这个功能技术上可行，但需要确认数据获取方式"
```

### 2. 让用户成为"数据源专家"

Agent 不知道用户能访问什么数据，必须询问：

```
"我无法知道您的内部系统是否开放接口，
 需要您确认以下问题..."
```

### 3. Demo 即契约

用户在 Demo 中做出的选择，成为后续开发的约束：

```
用户在 Demo 中："我可以接受手动导出 CSV"
DDDW 阶段：基于"手动导出"方案进行设计
```

## Risk Classification (Agent-Only Context)

| 风险类型 | Agent 可评估 | 处理方式 |
|---------|-------------|---------|
| 技术可行性 | ✅ 通过 WebSearch | 直接给出判断 |
| API 开放状态 | ⚠️ 需搜索确认 | 查询后告知用户 |
| 用户数据权限 | ❌ 不可知 | 交互式询问 |
| 商业授权 | ❌ 不可知 | 提示用户确认 |
| 反爬/封禁风险 | ⚠️ 部分可知 | 标注假设 |
| 长期稳定性 | ❌ 不可预测 | 设置用户预期 |

## Personal Notes

- 这个概念承认了 Agent 的能力边界，不是缺陷，是设计约束
- 关键转变：从"Agent 替用户判断"到"Agent 引导用户自我确认"
- 交互式风险暴露是核心技术——让用户在体验中意识到约束
- 可能需要建立一个"数据源清单"知识库，常见平台的数据获取方式
