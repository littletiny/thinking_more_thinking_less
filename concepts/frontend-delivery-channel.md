# Frontend Delivery Channel / 前端交付渠道

## One Sentence
用户必须拥有友好的前端界面，在从简原则下优先利用现有 IM 平台（微信/飞书/钉钉）作为输入输出载体，仅在 NLP 无法承载的复杂交互场景才考虑独立小程序或 App。

---

## Why It Matters

### 用户接受的底线

```
❌ 不可接受：
- CLI（命令行）
- TUI（终端界面）
- 浏览器展示 JSON
- 配置文件编辑

✅ 必须提供：
- 可视化界面
- 符合日常使用习惯
- 零学习成本
- 随时可用
```

### 为什么 IM 是最优解（从简视角）

| 维度 | 独立 App | 小程序 | IM 机器人 |
|------|---------|--------|----------|
| **开发成本** | 高（需原生或跨平台） | 中（需适配平台） | 低（API 调用） |
| **运维成本** | 高（服务器+IP+备案） | 低（平台托管） | 极低（平台托管） |
| **用户获取** | 难（需下载安装） | 中（需搜索/扫码） | 易（已是好友） |
| **使用频率** | 中（需主动打开） | 中（需主动打开） | 高（常用 IM） |
| **通知触达** | 需推送权限 | 需订阅 | 天然支持 |
| **信息密度** | 高 | 高 | 低（适合简单交互） |

**结论**：对于 DIY 个人工具，IM 机器人是成本最低、用户门槛最低的选择。

## Core Mechanism

### 渠道选择决策树

```
用户需求
    ↓
┌─────────────────────────────────────────┐
│ Step 1: 纯 NLP 能否承载？                │
│ （仅通过对话能否完成交互）                │
└──────────────────┬──────────────────────┘
                   │
        ┌──────────┴──────────┐
        ↓                     ↓
       能                    不能
        │                     │
        ↓                     ↓
   IM 机器人             需要结构化界面
   （微信/飞书/钉钉）         │
                        ┌────┴────┐
                        ↓         ↓
                  复杂表单    可视化操作
                  流程步骤    地图/图表等
                        │         │
                        └────┬────┘
                             ↓
                   ┌───────────────────┐
                   │ Step 2: 成本评估   │
                   ├───────────────────┤
                   │                   │
                   │ 使用频率高？      │
                   │ 交互复杂度高？    │
                   │ 长期使用？        │
                   │                   │
                   │ 是 → 小程序/App   │
                   │ 否 → IM + 链接    │
                   │                   │
                   └───────────────────┘
```

### 渠道分层策略

```
Layer 1: IM 机器人（默认）
├─ 适用：80% 的日常场景
├─ 形式：文字对话 + 简单卡片
├─ 平台：微信、飞书、钉钉
├─ 成本：极低（API 调用）
└─ 示例：
   用户："记录午餐 35 元"
   机器人："已记录：午餐 ¥35，本月餐饮累计 ¥1,250"

Layer 2: IM + Webview（过渡）
├─ 适用：需要结构化展示但不频繁
├─ 形式：IM 中发送链接，打开 Web 页面
├─ 成本：低（纯前端托管）
└─ 示例：
   用户："查看本月报表"
   机器人："[点击查看详细报表]" → 打开 Web 图表页

Layer 3: 小程序（复杂交互）
├─ 适用：高频 + 复杂交互
├─ 形式：微信小程序、飞书应用
├─ 成本：中（需开发+审核）
└─ 示例：
   - 多步骤表单填写
   - 实时数据可视化
   - 地图定位选择

Layer 4: 独立 App（极少）
├─ 适用：极高频 + 重交互 + 长期产品
├─ 形式：iOS/Android App
├─ 成本：高（开发+运维+推广）
└─ 适用场景：几乎不适用 DIY 个人工具
```

## Channel Comparison

### IM 机器人（推荐默认）

**优势**：
- 用户已在常用 IM 中，零安装
- 开发简单，调用平台 API 即可
- 通知天然送达
- 适合 NLP 交互

**局限**：
- 信息展示密度低
- 不适合复杂表单
- 不适合可视化操作

**适用场景**：
- 记账、待办、提醒
- 查询、简单记录
- 通知类功能

### 小程序（必要时的升级）

**优势**：
- 相对独立的体验
- 支持复杂交互
- 平台生态支持

**成本**：
- 开发：需前端技能（HTML/CSS/JS）
- 审核：需平台审核
- 运维：低（平台托管）

**适用场景**：
- 数据可视化（图表）
- 多步骤流程
- 需要用户频繁操作的功能

### Webview（折中方案）

**实现**：
- IM 中发送 H5 链接
- 用户点击打开浏览器
- 纯前端页面，可托管在 GitHub Pages/Netlify

**优势**：
- 无审核
- 成本低
- 灵活度高

**劣势**：
- 体验不如小程序
- 需要公网托管（但可用免费服务）

## Cost Analysis

### 模型推理成本 vs 渠道成本

```
用户担心的："模型推理是不是太贵？"

实际分析：
- GPT-4 级别：$0.03/1K tokens
- 一个对话回合：~500 tokens = $0.015
- 每天 50 次交互：$0.75/天 = ~$23/月

对比渠道成本：
- IM 机器人：几乎免费（平台 API）
- 小程序：免费（开发成本除外）
- 独立 App：服务器 $5-20/月 + IP/备案

结论：
对于个人工具，模型推理成本可接受
渠道成本优先选 IM（几乎零成本）
```

### 从简原则下的推荐

```
Phase 1: IM 机器人（MVP）
- 验证功能价值
- 成本最低
- 最快交付

Phase 2: IM + Webview（增强）
- 需要图表展示时
- 发送链接即可
- 无额外成本

Phase 3: 小程序（进阶）
- 高频使用且 IM 体验差
- 有开发余力
- 长期使用价值
```

## Key Principles

### 1. 渐进增强

```
从简开始：
IM 机器人 → IM + Webview → 小程序 → （极少）App

不要一上来就做小程序
先用 IM 验证价值
```

### 2. 平台跟随用户

```
用户常用什么 IM，就用什么平台
- 国内用户：微信优先
- 企业场景：飞书/钉钉
- 海外用户：WhatsApp/Telegram
```

### 3. NLP First

```
能用对话解决的，不用界面
- 记录："记一笔午餐 35" > 打开表单
- 查询："本月花了多少" > 打开报表
- 提醒："明天提醒我开会" > 设置界面

仅当 NLP 确实不便时才用界面：
- 选择日期（日历组件）
- 数据可视化（图表）
- 复杂配置（多字段表单）
```

## Personal Notes

- 前端渠道选择是产品体验的关键，但容易被技术人忽视
- 对于 DIY 工具，IM 机器人可能是被低估的方案
- OpenClaw 等社区工具验证了 IM 作为交互载体的可行性
- 需要警惕"为了技术而技术"——独立 App 的诱惑
- 核心原则：用户在哪儿，就在哪儿交付
