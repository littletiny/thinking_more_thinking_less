# Requirement Clarification Layer / 需求澄清层

## One Sentence
在"想法"和"实现"之间插入一个独立的验证层，用可丢弃的原型快速澄清需求，避免生产阶段的返工。

---

## Why It Matters

### 传统工具链的断裂

```
现有工具定位：

想法 ──────→ 设计/代码 ──────→ 产品
     ↑         ↑              ↑
   口头描述   Figma/v0.dev    上线

缺失的环节：
• 想法 → 设计：需要设计师翻译，业务方无法直接验证
• 设计 → 代码：设计稿不能运行，开发者理解有偏差
• 整个过程：没有记录"为什么这样设计"的决策历史
```

### 需求澄清层的价值

```
插入新层后的流程：

想法 ──────→ 需求澄清层 ──────→ 实现
            (MockCraft)
              • 自然语言输入
              • 生成可交互原型（不可用于生产）
              • 快速迭代验证
              • 自动记录决策历史
              • 确认后导出到实现工具

关键差异：
• 产出物是"可丢弃的"，专门用于验证
• 非技术人员可直接操作（自然语言）
• 修改成本极低（秒级重新生成）
```

## Source Context

From conversation: [mockcraft-design-discussion](../conversations/2026-03-09-mockcraft-design-discussion.md)

**Key case**: 讨论 MockCraft 系统定位时，发现 AI UI 生成工具市场存在结构性空位——现有工具要么偏向"生产代码"（v0.dev），要么偏向"视觉设计"（Figma），缺少专门的"需求验证"层。

**Key insight**: 需求澄清不是"设计"也不是"开发"，而是独立的认知层。它的目标是消除"理解偏差"，而非产出生产就绪的代码或设计稿。

**Critical thinking**: 经过批判性审视，发现该定位面临"伪需求风险"和"产出物价值存疑"的挑战，但作为**开源方法论工具**具有独特价值。

│  • 生成可交互原型  │  • 记录决策历史      │

│                   │                     │

### 与现有工具的区别

| 维度 | Figma | v0.dev | **RCL** |
|------|-------|--------|---------|
| 目标用户 | 设计师 | 开发者 | **业务人员/PM** |
| 产出物性质 | 设计稿 | 生产代码 | **可丢弃原型** |
| 修改成本 | 中（调设计） | 高（改代码） | **极低（重新生成）** |
| 记录决策历史 | ❌ 无 | ❌ 无 | **✅ 自动生成** |
| 可行性检查 | ❌ 无 | ❌ 运行时暴露 | **✅ 前置检查** |

## Critical Challenges

### 1. 伪需求风险

```
假设：用户需要"澄清需求"

现实可能是：
• 用户更想要"快速看到成果"而非"澄清"
• 迭代过程可能让用户感到挫败（"怎么还不对"）
• 如果最终仍需开发评估可行性，中间层价值被削弱

应对：
• 明确边界：RCL 不负责"实现"，只负责"对齐理解"
• 降低迭代成本：响应时间 < 3 秒，用户无感知等待
• 正向反馈：展示"已澄清 X 个模糊点"，给予成就感
```

### 2. 产出物价值存疑

```
问题：
• 开发者真的会看"迭代历史"吗？
• 生成的"需求文档"有人读吗？
• 可交互 Demo 验证后就被丢弃，是否浪费？

开源场景下的价值重构：
• 不再是"商业产品"，而是"方法论载体"
• 教育价值：演示从模糊到清晰的过程
• 研究价值：需求工程的实验工具
• 集成价值：被其他开源项目（低代码平台）集成
```

### 3. 用户角色错位

```
假设用户：业务人员/非技术用户

真实使用者可能是：
• 外包公司（减少返工）
• 咨询顾问（快速提案）
• 教育者（教学演示）
• 独立开发者（自用工具）

开源优势：
• 不需要"大众市场"，服务好细分群体即可
• 用户自备 API Key，解决成本问题
• 纯前端/可自托管，解决隐私问题
```

## Open Source Re-framing

### 从商业到开源的价值转换

```
商业产品逻辑（高风险）：
┌─────────────────────────────────────────┐
│  需要：大量付费用户                      │
│  成本：LLM 调用、服务器、团队工资         │
│  壁垒：功能、品牌、网络效应               │
│  成功标准：收入、增长、利润               │
└─────────────────────────────────────────┘

开源项目逻辑（可持续）：
┌─────────────────────────────────────────┐
│  需要：活跃的贡献者社区                  │
│  成本：维护者时间（可由社区分摊）          │
│  壁垒：方法论、标准、生态集成             │
│  成功标准：采用率、教育影响力、衍生项目    │
│  收益：简历、方法论验证、间接机会         │
└─────────────────────────────────────────┘
```

### 开源场景的独特价值

| 价值类型 | 说明 |
|---------|------|
 **方法论落地** | 让知识库的 mock-driven-validation 等理论有**可运行的实现** |
 **教育工具** | 用于教授"需求工程"，演示迭代过程 |
 **生态补位** | 填补开源工具链中"需求层"的空白（Penpot 是设计层，n8n 是执行层） |
 **私有化部署** | 企业内网使用，数据不出境 |
 **定制集成** | 被其他开源项目（如 AppSmith）集成为模块 |

## Key Principles

### 1. 可丢弃性（Disposability）

```
RCL 的核心设计哲学：

产出物是可丢弃的 → 用户没有心理负担 → 愿意试错
      ↓
修改成本极低 → 鼓励迭代 → 更快找到正确方向
      ↓
不替代生产工具 → 与 Figma/v0 互补而非竞争
```

### 2. 渐进式澄清（Progressive Clarification）

```
第一轮：捕捉意图（关键词提取）
第二轮：生成假设原型（快速验证方向）
第三轮：多场景扩展（覆盖边界情况）
第四轮：可行性检查（技术约束）
第五轮：生成交付物（PRD + Demo）

每轮都有明确的产出和验证标准
```

### 3. 过程即文档（Process as Documentation）

```
传统：先写 PRD → 再开发 → 文档与实际脱节

RCL：
迭代 1: "我要看板" → 生成 Demo A → 反馈"不对，要按区域"
迭代 2: "按区域分组的看板" → 生成 Demo B → 反馈"加上筛选"
迭代 3: "带筛选的分组看板" → 生成 Demo C → 确认"对！"

自动生成文档：
## 需求演进
- 初始：模糊的"看板"
- Iter1 发现：需要按区域分组（原需求未提及）
- Iter2 发现：需要筛选功能（交互细节）

## 最终确认
功能：分组看板 + 筛选器
数据：模拟 3 个区域、20 条记录
风险：无（无复杂技术约束）
```

## Personal Notes

- 经过批判性审视后，发现"需求澄清层"作为**商业产品**风险极高，但作为**开源方法论工具**价值明确
- 关键转变：从"卖工具"到"建立标准"，从"服务大众"到"教育/研究/集成"
- 与现有工具不是竞争关系，是**前置补充**——Figma 管"怎么做美"，v0 管"怎么实现"，RCL 管"做什么"
- 开源解决了最大的商业难题（成本、定价、用户教育），让项目可以专注于方法论本身的价值
