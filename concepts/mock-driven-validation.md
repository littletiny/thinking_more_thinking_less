# Mock-Driven Validation / Mock 驱动验证

## One Sentence
在 AI Code Agent 时代，Mock 从开发前的繁琐准备转变为快速验证用户主观感受的低成本工具，通过"生成-交互-反馈"的短循环让非开发者在触摸到具体成果后澄清真实需求。

---

## Why It Matters

传统瀑布模型的痛点——需求在早期难以完整定义——本质上是因为**非开发者只能"看到才理解"**。但传统 Mock 成本太高（需要开发者写代码），导致 Mock 范围尽可能小，用户反馈滞后。

AI Code Agent 改变了成本结构：
- 代码生成不再是瓶颈
- Mock 可以"即用即抛"
- 失败（用户说"这不是我想要的"）的成本趋近于零

这使得我们可以构建一个**更激进的验证循环**：在正式开发前，先用 Mock 让用户"触摸"到可能性空间，在交互中发现真实需求。

## Core Mechanism

### 传统模式 vs Mock-Driven 模式

| 维度 | 传统 Mock | Mock-Driven Validation |
|------|-----------|------------------------|
| **成本** | 高（人工编写） | 低（AI 生成） |
| **范围** | 最小化（减少浪费） | 最大化（探索可能性） |
| **目的** | 技术联调 | 需求验证 |
| **失败含义** | 返工、延期 | 预期内的学习 |
| **用户参与** | 被动接受设计 | 主动探索发现 |

### 三阶段循环

```
┌─────────────────────────────────────────────────────────┐
│  Intent Capture（意图捕获）                              │
│  └─ 用户用自然语言描述"我想要一个能...的东西"            │
│  └─ 不追求完美描述，接受模糊和矛盾                        │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│  Mock Generation（Mock 生成）                            │
│  └─ AI 基于意图生成可交互原型（而非文档）                 │
│  └─ 使用轻量级前端（Web 套壳即可）                        │
│  └─ 核心是"可触摸"，而非"功能完整"                       │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│  Subjective Calibration（主观校准）                      │
│  └─ 用户与 Mock 交互，产生"手感"                         │
│  └─ 关键问题："这是你想要的吗？哪里不对劲？"              │
│  └─ 收集反馈，提炼真实需求                                │
└──────────────────────┬──────────────────────────────────┘
                       │
        ┌──────────────┴──────────────┐
        │                             │
        ▼                             ▼
   ┌─────────┐                  ┌──────────┐
   | 确认命中  |                  | 需要调整  |
   └────┬────┘                  └────┬─────┘
        │                            │
        ▼                            ▼
   进入正式开发              回到 Intent Capture
   （此时需求已澄清）         （带着新的理解）
```

### 关键设计原则

1. **可丢弃性（Disposability）**
   - Mock 代码不需要维护
   - 甚至可以故意写"烂"代码（只要用户能感受到交互）
   - 目标是快速获得反馈，而非工程质量

2. **主观优先（Subjectivity First）**
   - 关注"感觉对不对"而非"逻辑通不通"
   - 接受用户说"我也说不清，但就是觉得怪"
   - 通过多轮迭代逼近"对的感觉"

3. **低成本分支（Low-Cost Branching）**
   - 可以同时生成 2-3 个不同风格的 Mock
   - 让用户 A/B 测试自己的偏好
   - "你更喜欢 A 还是 B？"比"你想要什么？"更容易回答

## Relationship with CREW

- **CREW**: 通过结构化对话澄清需求（"想清楚再动手"）
- **Mock-Driven Validation**: 通过可触摸成果验证需求（"先动手再理解"）

两者可以互补：
- 简单/直觉型需求 → 直接用 Mock-Driven
- 复杂/系统性需求 → 先 CREW 探索，再 Mock 验证关键假设

## Personal Notes

- 这个概念的核心是**成本结构变化带来的范式转移**
- 需要警惕：Mock 太"真实"可能导致用户误以为功能已完成
- 视觉/交互设计可能需要预设一些模板，否则生成成本还是高
- 可能的边界：对于需要复杂数据/后端逻辑的功能，Mock 能验证什么？
