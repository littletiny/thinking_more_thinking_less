# User Knowledge Reality / 用户知识现实

## One Sentence
普通用户没有数字化的领域知识库，其知识仅来自"上网查"和"个人经验"，因此在设计交互流程时不应假设用户能提供专业级的风险评估或技术判断。

---

## Why It Matters

### 错误的假设

许多 AI 产品设计隐含假设：
- 用户能清晰描述需求
- 用户能评估技术风险
- 用户有结构化的领域知识
- 用户能回答专业问题清单

### 现实

| 假设 | 现实 |
|------|------|
| 用户有知识库 | ❌ 只有散落各处的笔记和聊天记录 |
| 用户能评估风险 | ❌ 甚至不清楚风险存在 |
| 用户能回答技术问题 | ❌ 不专业，无法判断 |
| 用户有耐心填写清单 | ❌ 清单太长就放弃 |

**用户的实际知识来源**:
1. **上网查** —— 搜索引擎，零散信息
2. **脑子里** —— 个人经验，非结构化
3. **聊天记录** —— 更零散，难以检索

## Core Implications

### 1. 交互设计原则：低认知负荷

```
❌ 错误做法：
"请评估以下风险：
- 技术可行性风险 □ 高 □ 中 □ 低
- 数据获取风险 □ 高 □ 中 □ 低
- 合规风险 □ 高 □ 中 □ 低
..."

✅ 正确做法：
"这个功能需要获取您的微信数据，
 我们查了一下，微信不提供直接导出。
 
 您方便提供数据吗？
 [我可以导出聊天记录] [我没有这个数据]"
```

### 2. 风险评估的责任转移

```
传统：Agent 询问 → 用户评估 → 用户回答
        ↑______________|
        （用户不知道，无法回答）

新范式：Agent 评估 → 给出结论 → 用户选择
        ↑______________|
        （Agent 基于搜索做判断，用户只需做简单选择）
```

### 3. 问题类型重新分类

| 问题类型 | 能否问用户 | 处理方式 |
|---------|-----------|---------|
| **事实性问题** | ❌ 不能 | Agent WebSearch 自行解决 |
| **个人状态** | ✅ 能 | "您有 API 密钥吗？" |
| **偏好选择** | ✅ 能 | "A 和 B 您更喜欢哪个？" |
| **风险判断** | ❌ 不能 | Agent 判断后给出建议 |

## Integration with Workflow

### 修改后的可行性评估

```
用户提出需求
    ↓
Agent 内部评估（WebSearch，不问用户）
    ↓
┌─────────────────────────────────────────┐
│ 系统内部分类（不让用户感知复杂性）        │
├─────────────────────────────────────────┤
│                                         │
│  类别 A: 明确可行                        │
│  → 直接生成 Demo                         │
│                                         │
│  类别 B: 需要确认个人状态                 │
│  → 问简单问题："您有 XX 吗？"             │
│                                         │
│  类别 C: 存在硬性约束                    │
│  → 直接告知："这个功能无法实现，因为..."   │
│                                         │
│  类别 D: 需要权衡                        │
│  → 给出简化选项："完整版需要 XX，         │
│    简化版不需要，您选哪个？"              │
│                                         │
└─────────────────────────────────────────┘
    ↓
基于分类，设计最简交互
    ↓
Demo（不给用户负担）
```

## Key Principles

### 1. Don't Make Me Think（别让我思考）

用户不应该需要：
- 理解技术术语
- 评估风险等级
- 判断可行性
- 填写复杂表单

### 2. 结论前置

```
❌ "我们需要评估几个维度..."
✅ "这个功能可以做，但有个前提..."
```

### 3. 二元选择优先

```
❌ 多选、评分、开放式问题
✅ 是/否、A/B、接受/拒绝
```

### 4. 智能默认

```
"我们推荐方案 A（适合大多数用户），
 您也可以选方案 B（有特殊需求）"
```

## Personal Notes

- 这是典型的"专家盲区"——设计者懂技术，就假设用户也懂
- 关键认知：**用户的认知资源是稀缺品，必须精打细算**
- 一个经验法则：如果一个问题需要超过 10 秒思考，就不应该问
- 未来的优化方向：从用户历史行为推断偏好，减少询问
