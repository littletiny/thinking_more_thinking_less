# Hard Constraints System / 硬性约束系统

## One Sentence
系统应内置清晰、强硬的不可为清单，在技术可行性评估阶段自动拦截，无需用户参与判断，直接告知限制并提供替代方向。

---

## Why It Matters

### 混乱的风险沟通

```
❌ 模糊的风险描述：
"这个功能有一定风险，可能涉及隐私问题，
 建议您谨慎考虑，或者咨询专业人士..."
 
用户反应："有风险？那我还要不要做？什么风险？"
```

### 强硬约束的价值

```
✅ 清晰的不可为：
"系统无法帮您实现：
❌ 破解他人账号密码
❌ 绕过平台安全机制抓取数据  
❌ 自动生成虚假身份信息

这是硬性限制，不是技术问题。"

用户反应："明白，这个功能确实不能做"
```

## Core Mechanism

### 三层约束体系

```
┌─────────────────────────────────────────────────────────────┐
│                   Layer 1: Absolute No                      │
│                   绝对禁止（安全/法律）                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  系统内置，无需评估，直接拒绝                                │
│                                                             │
│  示例：                                                      │
│  - 破解/入侵系统                                            │
│  - 生成恶意软件                                             │
│  - 自动创建虚假身份                                         │
│  - 绕过付费机制                                             │
│  - 侵犯隐私的数据抓取                                       │
│                                                             │
│  响应方式："这个功能无法实现，因为这涉及..."                 │
│           （不提供替代方案，明确拒绝）                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                   Layer 2: Hard Limits                      │
│                   硬性限制（技术/平台）                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  系统评估后判定不可行，提供替代方向                          │
│                                                             │
│  示例：                                                      │
│  - 微信聊天记录直接读取（技术不可行）                        │
│  - 实时访问银行账户（API 不开放）                           │
│  - iOS 系统级功能（沙盒限制）                               │
│  - 需要特定硬件（用户没有）                                 │
│                                                             │
│  响应方式："直接实现不可行，但我们可以..."                   │
│           （提供降级方案或替代路径）                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                   Layer 3: Negotiable                       │
│                   可协商约束（成本/妥协）                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  可行，但有约束，需要用户选择                                │
│                                                             │
│  示例：                                                      │
│  - 功能 A 需要每月 $50 的 API 费用                          │
│  - 实时同步需要您每天手动触发                               │
│  - 完整功能需要您提供 API 密钥                              │
│                                                             │
│  响应方式："可以做，但需要您..."                            │
│           （用户选择接受或放弃）                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 约束判定流程

```
用户请求
    ↓
┌─────────────────────────────────────────┐
│ 1. Absolute No 检查                      │
│    （系统内置黑名单）                     │
└──────────────────┬──────────────────────┘
                   │
         ┌─────────┴──────────┐
         ↓                    ↓
      命中黑名单            未命中
         │                    │
         ↓                    ↓
    直接拒绝              继续评估
    （不提供替代）            │
                            ↓
                   ┌─────────────────┐
                   │ 2. Agent 评估    │
                   │    （WebSearch） │
                   └────────┬────────┘
                            │
                  ┌─────────┼─────────┐
                  ↓         ↓         ↓
               硬限制    可协商     明确可行
                  │         │         │
                  ↓         ↓         ↓
              提供替代    询问用户    直接 Demo
```

## Hard Limits 知识库（示例）

### 数据获取类

| 目标 | 状态 | 原因 | 替代方案 |
|------|------|------|---------|
| 微信聊天记录 | ❌ 硬限制 | 加密存储，无 API | 手动导出/仅新数据 |
| 抖音浏览历史 | ❌ 硬限制 | 平台封闭，反爬严格 | 用户主动记录 |
| 银行账户流水 | ❌ 硬限制 | 金融监管，API 受限 | 手动上传/开放银行（部分） |
| 淘宝订单数据 | ⚠️ 协商 | 需用户主动导出 | 提供导出教程 |
| Gmail 邮件 | ✅ 可行 | 提供 API | 需用户授权 |

### 平台限制类

| 功能 | iOS | Android | Web | 说明 |
|------|-----|---------|-----|------|
| 后台持续运行 | ❌ | ⚠️ | ✅ | iOS 严格限制 |
| 系统级按键监听 | ❌ | ❌ | ✅ | 安全限制 |
| 访问其他 App 数据 | ❌ | ❌ | N/A | 沙盒限制 |
| 推送通知 | ✅ | ✅ | ⚠️ | Web 需用户授权 |

### 成本约束类

| 功能 | 成本 | 说明 |
|------|------|------|
| GPT-4 级推理 | $0.03/1K tokens | 高频使用成本高 |
| 实时语音识别 | $0.006/15秒 | 持续监听昂贵 |
| 图像生成 | $0.02/张 | 批量生成成本高 |

## Interaction Design

### Absolute No 响应模板

```
用户："帮我写一个程序破解 WiFi 密码"

Agent：
"我无法协助实现这个功能。

这是硬性限制，不是技术问题：
• 未经授权访问网络是违法行为
• 系统设计明确禁止此类请求

建议您：
• 联系网络管理员获取密码
• 使用您有权访问的网络"

[结束对话，不提供替代]
```

### Hard Limit 响应模板

```
用户："我想自动同步我的微信聊天记录"

Agent：
"直接实现这个功能存在硬性限制：

❌ 微信不开放聊天记录 API
❌ 本地数据加密，无法直接读取

但我们有几个替代方案：

方案 A: 仅同步新消息
• 使用桌面微信的转发功能
• 将重要对话手动转发到您的邮箱
• 系统自动处理邮件内容

方案 B: 从导出文件导入  
• 微信支持导出聊天记录（.bak 格式）
• 您导出后，系统可以解析分析
• 适合一次性历史数据迁移

方案 C: 自建记录习惯
• 重要的聊天内容您手动复制粘贴
• 系统提供快捷记录工具
• 适合养成个人知识管理习惯

您倾向哪种方案？"
```

### Negotiable 响应模板

```
用户："我想要一个实时语音转文字的功能"

Agent：
"这个功能可以实现，但需要确认一个前提：

实时语音识别需要调用云服务，
预计每月使用成本约 $10-20（取决于使用量）。

或者，您可以选择：
• 离线版：准确率较低，但免费
• 批量版：非实时，先录音后转写，成本更低

您希望：
[接受实时版，支付云费用] 
[选择离线版]
[选择批量版]"
```

## System Architecture

```
Constraint Engine
├── Absolute No Module
│   ├── 安全黑名单（破解、攻击等）
│   ├── 法律红线（隐私、版权等）
│   └── 平台禁令（绕过机制等）
│
├── Hard Limits Knowledge Base
│   ├── 数据获取限制（各平台状态）
│   ├── 平台技术限制（iOS/Android/Web）
│   ├── 成本基准（各服务定价）
│   └── API 开放状态（常见服务）
│
└── Negotiable Parameters
    ├── 成本阈值
    ├── 性能权衡
    └── 用户偏好学习
```

## Personal Notes

- 硬性约束系统的核心价值：**减少用户认知负担，同时保护系统安全**
- 需要持续维护 Hard Limits Knowledge Base（各平台 API 状态变化）
- 平衡：过于严格的约束会限制创新，过于宽松会增加风险
- 透明性很重要：用户应该知道"为什么不可以"
