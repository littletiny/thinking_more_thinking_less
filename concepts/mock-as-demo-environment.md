# Mock as Demo Environment / Mock 即演示环境

## One Sentence
Mock 的本质是演示环境——通过可交互的原型引导用户逐步澄清需求，而非仅仅作为技术测试工具。

---

## Why It Matters

### 传统 Mock 的局限

```
传统 Mock：
目的：在技术开发前模拟 API，供前后端联调
范围：特定接口，精确返回值
用户：开发者

问题：
- 用户（非程序员）看不懂接口文档
- 无法引导用户思考"这个功能真的是我想要的吗"
- 静态的，不能随着用户反馈快速调整
```

### Demo Environment 的范式

```
Mock as Demo：
目的：让用户"体验"功能，在交互中澄清需求
范围：完整用户流程，包含模拟数据
用户：最终用户（非程序员）

优势：
- 用户能"触摸"到功能
- 通过交互发现"这里不对"
- 快速迭代调整
```

## Core Mechanism

### 从"技术 Mock"到"演示 Mock"

| 维度 | 技术 Mock | 演示 Mock |
|------|----------|-----------|
| **目的** | 接口联调 | 需求澄清 |
| **受众** | 开发者 | 最终用户 |
| **交互性** | 被动响应 | 主动引导 |
| **数据** | 精确、一致 | 丰富、场景化 |
| **迭代** | 慢（需改代码） | 快（NL 描述即可） |

### 演示 Mock 的设计原则

```
1. 场景化数据
   ❌ { "user_id": 1, "status": "active" }
   ✅ { "user": "张三", "最近活动": "昨天购买了咖啡豆" }

2. 完整流程
   不只是单个接口，而是完整用户旅程
   例如：搜索 → 浏览 → 详情 → 下单 → 支付

3. 可探索性
   用户可以尝试不同操作，看到不同结果
   模拟真实世界的多样性

4. 快速调整
   用户说"这里不对"→ 改 NL 描述 → 立即更新 Demo
```

## Key Principles

### 1. 最大化 Mock 覆盖率

```
原则：尽可能让用户在 Mock 阶段暴露需求

策略：
- 不只是核心功能，展示周边流程
- 模拟异常场景（"如果网络断了会怎样？"）
- 展示数据统计（"你想看到什么样的报表？"）
- 模拟提醒通知（"你希望什么时候收到提醒？"）
```

### 2. 数据即故事

```
❌ 使用抽象数据
   { "value": 100, "type": "A" }

✅ 使用场景化数据
   { "商品": "咖啡豆", "价格": "¥85", 
     "场景": "周末在家手冲" }
   
好处：
- 用户更容易产生共鸣
- 暴露隐含需求（"哦，我还需要记录购买场景"）
```

### 3. 错误即机会

```
用户操作 Demo "出错"（模拟的）
    ↓
暴露对错误的预期
    ↓
用户："这时候应该提示我重新输入"
    ↓
记录为需求的一部分
```

## Integration with NL-Mock

```
自然语言描述 Mock：

api: get_expense_records
behavior: >
  返回最近10条记账记录，包含：
  - 金额（带正负号，支出为负）
  - 类别（餐饮/交通/购物等）
  - 花费人（我/女朋友）
  - 时间（相对时间，如"昨天"）
  - 简短描述（场景化，如"周末咖啡"）
  
  如果是月初，包含上月对比提示
  如"比上月同期多花了 15%"

demo_purpose: "让用户看到记账后的数据统计效果"
```

## Success Metrics

### 好的演示 Mock

- ✅ 用户在 3 分钟内理解功能
- ✅ 用户主动提出"如果...会怎样"
- ✅ 发现至少一个初始假设之外的需求
- ✅ 用户可以说出"对，这就是我要的"或"这里需要改"

## Personal Notes

- Mock 作为演示环境，本质上是"视觉化的需求访谈"
- 好的 Demo 应该像"故事板"，让用户看到自己使用产品的场景
- 数据的选择很关键——抽象数据无法激发用户反馈，场景化数据可以
- 快速迭代能力（NL-Mock 支持）是这个模式的核心
