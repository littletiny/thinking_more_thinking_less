# Model-Code Agent Separation / 模型与 Code Agent 分工

## One Sentence
在 DIY 开发模式中，模型负责文本处理和交互流程（正确性不重要），Code Agent 负责数值计算（准确性关键），严格分离以避免重新实现整个系统。

---

## Why It Matters

### 能力边界

| 能力 | LLM 模型 | Code Agent |
|------|---------|-----------|
| **文本理解** | ✅ 优秀 | ⚠️ 需要解析 |
| **文本生成** | ✅ 优秀 | ❌ 不擅长 |
| **数值计算** | ❌ 易出错 | ✅ 精确 |
| **逻辑执行** | ⚠️ 不稳定 | ✅ 确定性 |
| **系统构建** | ⚠️ 知识截止 | ✅ 可搜索 |

### 全用 Code Agent 的问题

```
如果让 Code Agent 做所有事：
- 用户输入 → Code Agent 解析意图
- Code Agent 设计架构
- Code Agent 编写所有代码
- Code Agent 处理文本
- Code Agent 生成回复

结果：
- 实际上就是重新实现一个系统
- 复杂度爆炸
- 失去了快速拼凑的优势
```

### 全用模型的问题

```
如果让模型做所有事：
- 数值计算不准确（汇率、统计、财务计算）
- 逻辑执行不稳定（时好时坏）
- 无法与外部系统可靠交互

结果：
- 数据错误导致功能不可用
- 用户失去信任
```

## Core Mechanism

### 分工原则

```
┌─────────────────────────────────────────────────────────────┐
│                      LLM 模型                               │
│                  （意图理解 + 交互）                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 负责：                                                   │
│  - 理解用户自然语言输入                                     │
│  - 管理对话流程和上下文                                     │
│  - 生成文本回复和解释                                       │
│  - 文本匹配和查找（模糊搜索）                               │
│  - 判断用户意图和情绪                                       │
│                                                             │
│  💡 特点：                                                   │
│  - 正确性不重要（大概对就行）                               │
│  - 重要的是交互流畅                                         │
│  - 可以接受模糊匹配                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 明确边界
                              │ （什么交给 Code Agent）
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    Code Agent                               │
│                  （数值计算 + 执行）                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ 负责：                                                   │
│  - 数值计算（财务、统计、汇率等）                           │
│  - 精确的逻辑判断                                           │
│  - 与外部 API 的可靠交互                                    │
│  - 数据验证和清洗                                           │
│  - 文件读写操作                                             │
│                                                             │
│  💡 特点：                                                   │
│  - 准确性是关键                                             │
│  - 结果是确定性的                                           │
│  - 范围严格限制（不做架构设计）                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 协作流程示例

```
用户："帮我计算一下这个月的餐饮支出"
    ↓
┌─────────────────────────────────────────┐
│ 模型：理解意图                           │
│ - 识别：查询请求                         │
│ - 提取：时间范围（本月）、类别（餐饮）   │
│ - 决定：需要调用数值计算                  │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│ Code Agent：执行计算                     │
│ - 从存储中读取记录                       │
│ - 筛选本月 + 餐饮类别                    │
│ - 精确计算总和                           │
│ - 返回：{ total: 1250.50, count: 23 }   │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│ 模型：生成回复                           │
│ - 格式化结果："本月餐饮支出 ¥1,250.50"   │
│ - 添加洞察："比上月多了 ¥200，主要是..." │
│ - 询问："需要查看详细明细吗？"           │
└─────────────────────────────────────────┘
```

### 避免重新实现系统的边界

```
Code Agent 不做：                      Code Agent 只做：
━━━━━━━━━━━━━━━━━━━                    ━━━━━━━━━━━━━━━━━
❌ 架构设计                            ✅ 简单的数值计算
❌ 复杂的状态管理                       ✅ 确定性的逻辑执行
❌ 自然语言理解                         ✅ 数据存取
❌ 用户意图判断                         ✅ 外部 API 调用
❌ 文本生成                             ✅ 数据验证

原则：
- Code Agent 是"计算器"+"执行器"
- 不是"架构师"+"设计师"
- 复杂的决策和交互由模型处理
```

## Architecture

```
User
  ↓
Model（入口）
  ├─→ 理解意图
  ├─→ 决定：需要计算/执行？
  │     ├─→ 是 → 调用 Code Agent
  │     │         ↓
  │     │     Code Agent
  │     │         - 数值计算
  │     │         - 数据操作
  │     │         - API 调用
  │     │         ↓
  │     └─→ 返回结果
  │
  ├─→ 生成回复（基于结果）
  ↓
User
```

## Examples

### 场景 1: 记账应用

```
Model 处理：
- "记录一笔午餐花费 35 块"
- 理解：添加支出，类别餐饮，金额 35
- 决定：需要存储

Code Agent 处理：
- 验证金额格式
- 存入数据库/文件
- 返回确认

Model 处理：
- "已记录：午餐 ¥35"
- "本月餐饮已花 ¥1,250"
```

### 场景 2: 文本查找

```
用户："找一下我之前记录的关于咖啡的笔记"

Model 处理（不调用 Code Agent）：
- 文本匹配（模糊搜索）
- 返回相似度高的结果
- 即使匹配不完全准确也没关系

（因为正确性不重要，交互流畅更重要）
```

### 场景 3: 复杂计算

```
用户："帮我算一下年化收益率"

Model：
- 理解需要计算
- 提取必要参数

Code Agent：
- 精确计算 IRR
- 返回数值结果

Model：
- 解释结果含义
- 生成友好的回复
```

## Key Principles

### 1. 正确性分级

```
高正确性要求（Code Agent）：
- 财务计算
- 数据统计
- 外部 API 交互
- 数据持久化

低正确性要求（Model）：
- 文本匹配
- 意图理解
- 回复生成
- 流程控制
```

### 2. 最小化 Code Agent 职责

```
原则：Code Agent 只做"不得不做"的事

不得不：
- 数值计算（模型做不好）
- 外部交互（模型无法直接做）

可以不做：
- 文本处理（模型更擅长）
- 决策判断（模型更灵活）
- 用户交互（模型更自然）
```

### 3. 失败处理

```
Code Agent 失败：
- 返回错误码
- 模型解释错误并建议

Model 失败（如理解错误）：
- 用户纠正
- 模型重新理解
- 不需要 Code Agent 介入
```

## Personal Notes

- 这个分工是务实的：各自发挥所长
- 模型的"模糊能力"是优势（文本匹配），不要强求精确
- Code Agent 的"精确能力"是优势（数值计算），不要用来做决策
- 关键是边界清晰，避免 Code Agent 膨胀成"重新实现系统"
