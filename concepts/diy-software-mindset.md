# DIY Software Mindset / DIY 软件思维

## One Sentence
面向非程序员用户的开发，核心范式从"制造工业标准件"转变为"个人 DIY 拼凑"——放弃代码质量和工程标准，追求简单、能用、无安全隐患即可。

---

## Why It Matters

### 工业开发 vs DIY 开发

| 维度 | 工业软件开发 | DIY 软件开发 |
|------|-------------|-------------|
| **目标用户** | 成千上万用户 | 往往只有作者自己 |
| **生命周期** | 数年甚至十年 | 可能只用几次 |
| **代码质量** | 重要（维护成本） | 不重要（用完即弃） |
| **架构设计** | 需要可扩展 | 不需要，能跑就行 |
| **性能优化** | 核心指标 | "PHP 是最好的语言" |
| **测试覆盖** | 必须 | 手动点一下没问题就行 |
| **工程规范** | 严格遵守 | 没有规范 |
| **技术债务** | 需要管理 | 债多不愁 |

### 核心洞察

> "我们的目的不是生成工业标准件，我们的目的是各种拼拼凑凑的 DIY"

对于非程序员用户：
- 他们不需要"写得好的代码"
- 他们需要"能解决问题的工具"
- 代码是手段，不是目的
- 能跑起来 > 跑得好
- 今天能用 > 永远能用

## Core Principles

### 1. 能用 > 好用 > 优雅

```
传统优先级：
代码优雅 → 架构清晰 → 性能优秀 → 功能完整 → 能用

DIY 优先级：
能用 → 功能完整 → （其他都不重要）
    ↓
如果"能用"需要牺牲优雅？毫不犹豫牺牲
```

### 2. 组装优先于制造

```
传统：
分析需求 → 设计架构 → 编写代码 → 测试优化
     ↑_________________________|
              （从零制造）

DIY：
找现成工具 → 胶水代码连接 → 能跑就行
     ↑________________|
        （拼凑组装）
```

**组装策略**:
- 能调 API 就不自己实现
- 能用开源库就不自己写
- 能复制粘贴就不重构
- 能硬编码就不配置化

### 3. 安全是唯一的硬约束

```
可以妥协的：        不能妥协的：
• 代码风格          • 安全漏洞
• 性能优化          • 数据泄露风险
• 可维护性          • 权限滥用
• 单元测试
• 文档
```

### 4. 一人项目假设

```
默认假设：
• 开发者 = 唯一用户
• 不需要协作
• 不需要交接
• 代码是"写给自己看的笔记"
• 今天能跑就行，不管明天
```

### DIY 模式下的可行性评估

```
传统评估：
"这个功能需要：
- 微服务架构
- 数据库设计
- API 设计
- 单元测试
- ..."

DIY 评估：
"这个功能：
- 能调现成的 API 吗？✅
- 需要存数据吗？硬编码/JSON 文件就行
- 有安全漏洞吗？检查一下
- 能跑吗？能 → 通过"
```

### DIY 组装工具箱

```
核心哲学：不要造轮子，找轮子

数据存储：
- 传统：数据库 + ORM
- DIY：JSON 文件、LocalStorage、甚至硬编码

后端服务：
- 传统：自建服务器 + 框架
- DIY：Serverless、云函数、甚至纯前端

第三方集成：
- 传统：官方 SDK + 封装
- DIY：直接调 HTTP、复制粘贴示例代码

UI：
- 传统：组件化 + 设计系统
- DIY：Bootstrap、现成的 Admin 模板、甚至无 UI（命令行）
```

## Risk Reassessment

### 在 DIY 模式下重新评估风险

| 传统风险 | DIY 视角 | 处理方式 |
|---------|---------|---------|
| 性能差 | 无所谓 | 忽略 |
| 代码难维护 | 无所谓 | 忽略 |
| 扩展性差 | 无所谓 | 忽略 |
| 技术债务 | 债多不愁 | 忽略 |
| 没有测试 | 手动测试 | 接受 |
| 架构不优雅 | 不重要 | 忽略 |
| 安全漏洞 | **关键** | 必须检查 |
| 数据泄露 | **关键** | 必须检查 |

### 安全底线的具体化

```
即使 DIY，也必须检查：

✅ 输入验证
   - 用户输入有没有直接拼接到 SQL/命令行
   - 有没有 XSS 风险

✅ 敏感信息
   - 密钥、密码有没有硬编码上传
   - 有没有暴露到前端

✅ 权限控制
   - 有没有越权访问风险
   - 文件操作有没有限制路径

✅ 外部调用
   - 调用的 API 是否可信
   - 有没有 SSRF 风险

其他一切：都可以妥协
```

## Success Criteria

### DIY 项目的完成标准

```
传统项目：
□ 代码 review 通过
□ 测试覆盖率 > 80%
□ 性能 benchmark 达标
□ 文档完整
□ 架构评审通过
□ ...

DIY 项目：
□ 能跑起来
□ 解决了我当前的问题
□ 没有安全漏洞
□ （其他都是可选）
```
