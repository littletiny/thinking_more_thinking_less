# Mock Feasibility Boundary / Mock 可行性边界

## One Sentence
并非所有接口都能被有效 Mock——必须在探索期建立技术可行性检查机制，确保 Mock 的行为在工程实现上是可达的，避免验证通过却无法交付的工程失败。

---

## Why It Matters

### 危险的陷阱

```
用户："我想要一个能实时翻译全球任何语言对话的 App"
    ↓
NL Mock: "接收音频流，返回实时翻译文本"
    ↓
Demo 运行良好（模拟数据）
    ↓
用户："对，这就是我要的！"
    ↓
DDDW 启动 → 工程团队开始实现
    ↓
发现：实时语音翻译需要...
  - 低延迟 ASR（需要特定云服务）
  - 专业级机器翻译 API（成本高）
  - 边缘计算部署（复杂）
    ↓
无法实现原 Demo 承诺的体验
    ↓
工程失败 / 需求重新谈判 / 信任破裂
```

**核心问题**: Mock 关注的是"行为模拟"，而非"实现可行性"。当两者脱节，就会出现"验证通过却无法交付"的灾难。

## Core Mechanism

### 可行性维度

| 维度 | 描述 | 典型失败案例 |
|------|------|-------------|
| **技术可行性** | 当前技术能否实现 | 实时100%准确率的通用翻译 |
| **成本可行性** | 实现成本是否可接受 | 每个请求调用 GPT-4 级模型 |
| **时间可行性** | 交付周期是否现实 | 3个月内实现自动驾驶 |
| **合规可行性** | 是否满足法律/政策 | 绕过平台限制的数据抓取 |
| **依赖可行性** | 外部依赖是否可控 | 依赖未公开 API 的第三方服务 |

### 三层防御机制

```
┌─────────────────────────────────────────────────────────────────┐
│                   Layer 1: Pre-Mock Check                       │

│     ┌──────────┐         ┌──────────┐                          │

## Integration with Workflow

### 修改后的 Exploration Phase

```
用户意图
    ↓
┌──────────────────────────────────────┐
│ Feasibility Pre-Check                │  ← 新增
│ - 快速评估技术可行性                 │
│ - 识别明显不可能的需求               │
└──────────────────┬───────────────────┘
                   │
        ┌──────────┴──────────┐
        ↓                     ↓
   高可行性               低可行性
        │                     │
        ↓                     ↓
   NL Mock 生成         与用户协商
        │              - 解释约束
        │              - 提出替代
        ↓                     ↓
   Demo 生成 ───────────────────┘
        │
        ↓
   用户验证
        │
        ↓
   ┌──────────────────────────────────────┐
   │ Pre-DDDW Feasibility Gate            │  ← 强化
   │                                      │
   │ 工程团队评估：                       │
   │ - 所有高风险 Mock 的实现方案         │
   │ - 成本与时间的现实性                 │
   │ - 外部依赖的可控性                   │
   │                                      │
   │ 如果不可行 → 返回 Exploration        │
   │ 如果可行   → 启动 DDDW               │
   └──────────────────────────────────────┘
```

## Key Patterns

### Pattern 1: Progressive Disclosure of Constraints

不要一开始就列举所有技术限制，而是在迭代中逐步揭示：

```
Iter 1: 用户要求"实时翻译"
       → Mock: 简单返回翻译文本
       → 用户："对，但翻译质量要像人工一样"

Iter 2: 更新 Mock 包含质量模拟
       → 用户："很好，但要支持50种语言"
       
Iter 3: 标注可行性风险
       → "支持50种语言，但小语种准确率较低，是否接受？"
       → 用户："可以接受主要语言优先"
       
Final: 明确标注的约束成为需求文档的一部分
```

### Pattern 2: Mock 分级

```yaml
mock_classification:
  safe:
    description: "技术成熟，实现可控"
    examples: ["CRUD 操作", "简单业务逻辑"]
    process: "直接 Mock，低风险"
    
  risky:
    description: "技术可行，但有挑战"
    examples: ["实时音视频", "大规模并发"]
    process: "标注风险，工程团队预评估"
    
  experimental:
    description: "技术前沿或不确定"
    examples: ["AGI 功能", "突破性创新"]
    process: "必须技术验证后再进入 DDDW"
```

### Pattern 3: Feasibility Spike

对于高风险 Mock，在 Exploration Phase 插入技术验证：

```
用户确认 Demo
    ↓
存在高风险 Mock？
    ↓
是 → Feasibility Spike (1-2天)
     - 快速验证关键技术假设
     - 构建概念验证原型
     - 评估实际性能/成本
    ↓
Spike 通过 → 继续 DDDW
Spike 失败 → 返回 Exploration，调整需求
```

## Anti-Patterns

### ❌ 盲目相信 Mock

```
错误："Demo 跑通了，用户确认了，一定能实现"
正确："Demo 验证了用户意图，但技术可行性需要独立评估"
```

### ❌ 过度承诺

```
错误：为了不打击用户热情，隐瞒技术风险
正确："这个功能很酷，但实现有挑战，我们来探讨可行方案"
```

### ❌ 过早优化

```
错误：在 Exploration Phase 就进行详细的技术设计
正确：Exploration 关注"能不能做"，不是"怎么做"
```

## Special Note: Agent-Only Context

在 [[agent-only-engineering]] 场景下，可行性边界需要特殊处理：

| 可行性类型 | 传统团队 | Agent-Only |
|-----------|---------|-----------|
| 技术可行性 | 工程师经验 | WebSearch + 知识库 |
| 数据可达性 | 工程师行业知识 | **必须在 Demo 中交互确认** |
| 权限状态 | 内部了解 | **必须由用户自我声明** |

**关键调整**: 可行性检查从"Agent 后台判断"转为"Demo 前台交互暴露"
