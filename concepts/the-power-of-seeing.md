# The Power of Seeing / 看见的能力

## One Sentence
人无法想象自己不知道的东西，必须通过"看见"（Demo/可视化/示例）才能驱动尝试和确认，这是低代码/无代码平台的核心价值所在。

---

## Why It Matters

### 传统方法的失败

```
询问用户需求：
"你想要什么功能？"
    ↓
用户："我不知道..."
    ↓
"你能描述一下你的需求吗？"
    ↓
用户："我想要一个能帮我管理事情的工具..."
    ↓
（过于抽象，无法落地）
```

**问题**：用户无法想象自己不知道的东西。

### 看见的力量

```
展示可能性：
"这是我能做的三种样子："
    ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
    │   方案 A    │  │   方案 B    │  │   方案 C    │
    │  [截图/Demo]│  │  [截图/Demo]│  │  [截图/Demo]│
    └─────────────┘  └─────────────┘  └─────────────┘
    ↓
用户："B 比较接近，但我想把这里改成..."
    ↓
（具体、可执行的反馈）
```

**突破**：一旦看见，用户就能判断、修改、确认。

## Core Mechanism

### 认知科学基础

```
人类的认知限制：
- 工作记忆容量有限（7±2 个信息块）
- 无法操作纯抽象概念
- 需要具体实例来理解

看见的作用：
- 将抽象需求转化为具体形式
- 提供讨论的锚点
- 降低认知负荷
- 激活用户的判断能力
```

### "看见"的层次

```
Level 1: 看见示例（Example）
"这是一个做好的记账应用"
用户："哦，原来是这样的"

Level 2: 看见可能性（Possibility）
"你可以改成这样、这样、或这样"
用户："那我想要 A+B 的组合"

Level 3: 看见过程（Process）
"看着我是怎么修改的"
用户："原来可以这样做，我也试试"

Level 4: 看见反馈（Feedback）
"改完后立即看到结果"
用户："不对，再调一下..."
```

### 与 Mock-Driven Validation 的关系

```
Mock-Driven Validation 的本质：
"用看得见的东西验证看不见的需求"

不是：
用户描述需求 → 开发 → 交付 → 发现不对 → 返工

而是：
快速展示可能性 → 用户看见 → 确认/修改 → 再展示 → 确认 → 开发

关键转变：
- 验证点前置（在开发前验证）
- 用户参与设计（看见后提出修改）
- 降低返工成本（Mock 成本低）
```

## Implications for Design

### 1. 展示优先于询问

```
❌ "你想要什么功能？"
✅ "这些功能你觉得哪个有用？"

❌ "你喜欢什么风格？"
✅ "这三种风格你选哪个？"

❌ "你的流程是什么样的？"
✅ "这个流程对吗？哪里不对？"
```

### 2. 多给选项，少给白纸

```
❌ 空白画布 + "创造吧"
✅ 半成品 + "修改吧"

低代码平台的成功因素：
- Webflow：从模板开始
- Notion：从模板开始
- Figma：从组件开始

失败因素：
- 给用户一张白纸
- 说"你可以创造任何东西"
- 用户："我不知道创造什么"
```

### 3. 即时反馈是关键

```
看见 → 修改 → 立即看见结果 → 再修改

这是：
- Figma 的成功（设计即见即所得）
- Excel 的成功（公式即时计算）
- Scratch 的成功（积木即时运行）

不是：
- 写代码 → 编译 → 运行 → 看结果（太慢）
```

## Comparison: 看见 vs 平台能力

| 维度 | 平台能力导向 | 看见能力导向 |
|------|-------------|-------------|
| **核心问题** | "我们能提供什么功能？" | "用户能看见什么可能性？" |
| **用户入口** | 功能列表 | 示例/模板 |
| **交互方式** | 配置参数 | 修改示例 |
| **学习曲线** | 需要理解平台概念 | 从模仿开始 |
| **成功率** | 低（用户不知道怎么开始） | 高（有起点） |

## Applications

### 在 DIY 系统中的应用

```
Exploration Phase：
- 第一步：展示 3 个 Mock 方案
- 第二步：用户选择并修改
- 第三步：确认 Demo
- 第四步：开发

关键：
- Mock 不是技术工具，是"看见的工具"
- NL-Mock 的价值：快速生成"可看见的可能性"
- Demo 的价值：让用户"看见"最终成果
```

### 在其他领域的应用

```
教育：
- 不是："学习编程语法"
- 而是："看着我做，然后你试试"

设计：
- 不是："你想设计什么？"
- 而是："从这些模板开始改"

写作：
- 不是："你想写什么？"
- 而是："这是大纲，你填内容"
```

## Key Insight

> **低代码平台的核心竞争力不是"降低了多少代码量"，而是"让多少看不见的东西变得可见"。**

```
技术能力（代码生成）是手段
看见能力（Demo/Mock/可视化）是目的

用户不因为"不用写代码"而买单
用户因为"我能看见我要什么"而买单
```

## Personal Notes

这个洞察改变了我的理解：
- 之前：DIY 系统的价值是"让非程序员能写代码"
- 现在：DIY 系统的价值是"让用户看见可能性"

写代码是手段，看见可能性才是目的。

这也解释了为什么：
- 纯代码生成工具（如 Copilot）只是辅助
- 可视化工具（如 Figma）改变行业
- 模板比空白画布受欢迎

看见 > 能力
