# Git & Agent

## 一句话
Git 为 Agent 的不可靠编辑提供可逆性和审计能力，是 Agent 工作的"容错基底"。

---

## 为什么重要

Agent 是不可靠的：
- 同样的 prompt，两次结果不同
- 会"幻觉"（生成不存在的内容）
- 上下文窗口有限，可能忽略关键信息

**没有 Git**：改坏了无法回滚，不敢让 Agent 做复杂任务。
**有 Git**：从"不可逆的冒险"变成"可回滚的实验"。

## 讨论与挣扎

### Git 的三重角色

1. **事务管理器**：Commit 是原子单位，失败可回滚
2. **审计证据链**：METIS/ECTM 要求"过程可复现"，Git 提供时间戳证据
3. **协作仲裁者**：多人/多 Agent 协作的基础设施

### 与 Obsidian 的关系

| Obsidian | Git + Agent |
|----------|-------------|
| 知识可视化、探索 | 版本控制、审计、协作 |
| 人阅读、探索 | Agent 编辑、维护 |
| 前端 | 后端基础设施 |

**互补关系**：人关注内容，Agent 维护结构，Git 保障安全。

### Git 只是备份吗？

**不是**。Git 提供：
- 可逆性（任何改动都可撤销）
- 原子性（每个 commit 是完整状态）
- 分支（并行实验）
- 审计（谁、什么时候、改了什么）

### 高频小 Commit

**Agent 专用策略**：每个文件独立 commit，失败只回滚该文件。

对人类太繁琐，但 Agent 没有情绪，不会累。

---

## 核心机制

**可逆性改变风险计算**：
- 无 Git：风险 = 破坏程度 × 修复时间（极高）
- 有 Git：风险 = 破坏程度 × 回滚时间（接近零）

**Git 是 Agent 的"免疫系统"**：
- 不阻止错误，但确保错误可恢复
- 让 Agent 可以大胆尝试

## 个人笔记

- Git 让"实验"成为可能：在 branch 上任意尝试，失败丢弃，成功合并
- Git 历史是知识的"化石记录"，可复盘、可学习
- Git 对非技术人员还是太难，可能需要更简单的"可逆性"方案
