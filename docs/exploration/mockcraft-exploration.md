## Round 1: Context Gathering

### 初始想法 (Initial Idea)

> 来自 sth.md 的核心定义：
> 
> **MockCraft** 是一个**自然语言驱动的需求验证与原型生成系统**，也可称为"可执行的需求文档"（Executable PRD）。
>
> 本质：让需求从"被阅读的文本"变成"被体验的界面"

### 触发背景

- 传统需求传递存在断裂：想法 → 设计/代码 → 产品
- 业务方无法直接验证想法，必须经过设计师/开发者翻译
- 需求传递中存在信息损耗和决策历史缺失

### 目标用户

| 用户类型 | 痛点 | 使用场景 |
|----------|------|----------|
| 业务人员/产品经理 | 说不清需求，无法直接验证想法 | 快速将模糊想法变成可交互原型 |
| 设计师 | 反复修改设计稿，需求不明确 | 接收已澄清的需求，做视觉优化 |
| 开发者 | 需求理解有偏差，返工成本高 | 基于确认的原型和 PRD 开发 |

### 成功标准

- 业务人员能用自然语言描述需求，5分钟内看到可交互 Demo
- 修改迭代成本趋近于零（秒级重新生成）
- 自动记录需求演进历史，生成 PRD

## Round 2: Problem Definition

### 问题陈述 (Problem Statement)

#### 5 Whys 分析

1. **为什么需求经常出错？** → 业务方口头描述模糊
2. **为什么描述模糊？** → 缺乏可视化的验证工具
3. **为什么没有验证工具？** → 现有工具要么太专业（Figma），要么直接生成代码（v0.dev）
4. **为什么这会造成问题？** → 业务方不敢试错，接受"差不多就行"
5. **根因是什么？** → **缺少专门的"需求验证层"**，业务人员没有低门槛的试错工具

### 核心痛点

| 环节 | 问题 | 后果 |
|------|------|------|
| 业务方→产品经理 | 口头描述模糊 | "我想要一个看板" |
| 产品经理→设计师 | 理解偏差 | 设计稿不是业务方想要的 |
| 设计师→开发者 | 交互细节缺失 | 开发成果与预期不符 |
| 开发→上线 | 技术约束未提前暴露 | 返工或妥协 |

### 不解决的后果

- 持续的需求返工，项目延期
- 业务方与研发团队摩擦增加
- 沉没成本越高，越难调整方向

## Round 3: Solution Brainstorming

### 替代方案分析

| 方案 | 优点 | 缺点 | 评估 |
|------|------|------|------|
| **A: 纯 Figma 插件** | 设计师熟悉，可视化强 | 业务方不会用，修改成本中 | Meh |
| **B: 直接生成 Coze/n8n 工作流** | 能直接运行 | 只能覆盖自动化场景，通用性差 | Meh |
| **C: 类 v0.dev 的代码生成** | 开发者喜欢 | 业务方看不懂代码，不解决"验证"问题 | Hate |
| **D: MockCraft (独立澄清层)** | 低门槛、可丢弃、记录历史 | 需要新建工具 | **Like** |

### 选择方向

**选定方案 D: MockCraft 作为独立的需求澄清层**

理由：
1. 填补市场空白（业务人员导向）
2. 与现有工具形成上下游关系而非竞争
3. 可丢弃原型的理念降低试错心理门槛

## Round 4: Deep Dive

### 系统架构

```
┌─────────────────────────────────────────┐
│  Layer 3: 实现层（Implementation）       │
│  • Figma - 高保真设计                    │
│  • v0.dev - 生产代码                     │
│  • Coze/n8n - 自动化工作流               │
└─────────────────────────────────────────┘
                    ▲
                    │ 导出确认后的需求
┌───────────────────┼─────────────────────┐
│  Layer 2: 澄清层   │  (MockCraft)       │
│  • 自然语言输入    │  • 快速迭代验证      │
│  • 生成可交互原型  │  • 记录决策历史      │
│  • 多场景覆盖      │  • 可行性检查        │
└─────────────────────────────────────────┘
                    │
┌───────────────────▼─────────────────────┐
│  Layer 1: 意图层（Intent）               │
│  • 业务人员的模糊想法                    │
│  • "我想要一个..."                      │
└─────────────────────────────────────────┘
```

### 核心组件

| 组件 | 职责 | 输入 | 输出 |
|------|------|------|------|
| **意图提取器** | 从自然语言提取关键词 | "我要一个看板" | `{type: "board", entities: ["task"], features: []}` |
| **原型生成器** | 生成可交互 Demo | 结构化意图 | HTML/CSS/JS 原型 |
| **场景扩展器** | 覆盖边界情况 | 基础原型 | 空状态/大数据/错误状态 |
| **可行性检查器** | 技术约束评估 | 原型配置 | 风险标记 + 建议 |
| **文档生成器** | 输出 PRD | 迭代历史 | Markdown PRD |

### 数据流

```
用户输入 → 意图提取 → 原型生成 → 用户反馈
                              ↑___________|
                                  ↓
                           确认后 → 文档生成 → 导出到下游工具
```

### 渐进式澄清流程

```
第一轮：捕捉意图（关键词提取）
   "我要一个看板" → 提取：看板、任务管理

第二轮：生成假设原型（快速验证方向）
   生成 Demo A → 用户反馈："不对，要按区域分组"

第三轮：多场景扩展（覆盖边界情况）
   按区域分组看板 → 模拟空数据/大量数据/错误状态

第四轮：可行性检查（技术约束）
   检查：是否需要实时协作？数据量多大？

第五轮：生成交付物（PRD + Demo）
   输出：确认的需求文档 + 可交互原型
```

## Round 5: Trade-off Analysis

### 关键权衡

| 维度 | 选项 A | 选项 B | 选择 | 理由 |
|------|--------|--------|------|------|
| **产品形态** | 独立 SaaS 产品 | 开源工具/方法论 | **开源工具** | 独立产品风险高，开源更有价值 |
| **技术栈** | 纯前端生成 | 后端渲染 | **纯前端** | 降低部署成本，保护用户隐私 |
| **原型保真度** | 低保真（线框） | 中保真（可交互） | **中保真** | 足够验证需求，不过度设计 |
| **输出格式** | 多种格式 | 单一标准格式 | **Markdown + HTML** | 通用、易读、易版本控制 |
| **集成方式** | 深度集成飞书/Coze | 通用导出接口 | **通用导出接口** | 不与特定平台绑定 |

### 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 用户不理解"可丢弃原型"理念 | 中 | 高 | 强调这是验证工具，不是最终产品 |
| 与 v0.dev/Figma AI 功能重叠 | 中 | 中 | 明确差异化：专注需求澄清而非生产 |
| 技术实现复杂度高 | 低 | 高 | 先实现核心意图提取 + 简单原型生成 |

### 约束条件

- **时间**: MVP 应在 2-4 周内可演示
- **技术**: 优先使用现有 LLM API，不自建模型
- **资源**: 单人或小型团队可维护

## Round 6: Consolidation

### 共识总结

1. **MockCraft 不是**另一个 Coze/n8n/v0.dev
2. **MockCraft 是**需求工程的方法论工具，填补"意图→实现"之间的空白
3. **最佳形态**: 开源项目，可嵌入现有生态的配置生成节点
4. **核心价值**: 让业务人员在投入昂贵开发资源前验证想法

### 明确的事项

✅ **清晰的部分**:
- 产品定位（需求澄清层）
- 目标用户（业务人员/PM）
- 核心机制（可丢弃原型 + 渐进澄清）
- 与现有工具的关系（上下游互补）
- 输出物（PRD + 可交互原型）

⚠️ **需要进一步确认的部分**:
- 技术实现细节（具体使用哪个框架/库）
- 原型生成的具体交互形式
- 与 LLM 的集成方式（API 选择）

### 下一步建议

1. **创建技术设计文档**: 确定前端框架、LLM 集成方案
2. **开发 MVP**: 实现最简单的意图提取 + 原型生成
