## 1. 概述

### 1.1 产品定位

MockCraft 是一个**自然语言驱动的需求验证与原型生成系统**，也可称为"可执行的需求文档"（Executable PRD）。

**核心定位**: 需求工程的"三明治结构"中的**需求澄清层**

```
上游：自然语言需求（用户头脑中的想法）
      ↓
中游：需求澄清层（MockCraft）★ 核心定位
      ↓
下游：生产代码/无代码工具（Coze/n8n/飞书/Figma）
```

### 1.2 目标用户

| 用户角色 | 核心诉求 | 使用频率 |
|----------|----------|----------|
| 业务人员/产品经理 | 快速验证想法，降低试错成本 | 高频 |
| 设计师 | 接收已澄清的需求，做视觉优化 | 中频 |
| 开发者 | 获取明确的需求文档和原型参考 | 中频 |

### 1.3 价值主张

> **让需求从"被阅读的文本"变成"被体验的界面"**

1. **需求澄清**: 帮"说不清需求的业务人员"把想法变成可交互 Demo
2. **降低门槛**: 非技术人员用自然语言描述 → 生成可用原型
3. **快速验证**: 5分钟生成 Demo，1分钟修改迭代
4. **智能决策**: 判断需求该用飞书（70%简单场景）还是 DDDW（30%复杂场景）

## 2. 问题陈述

### 2.1 现状痛点

```
传统流程的问题：

想法 ──────→ 设计/代码 ──────→ 产品
     ↑         ↑              ↑
   口头描述   Figma/v0.dev    上线
```

**断裂点**:
- **想法→设计**: 需要设计师翻译，业务方无法直接验证
- **设计→代码**: 设计稿不能运行，开发者理解有偏差
- **无决策记录**: 整个过程没有记录"为什么这样设计"的历史

### 2.2 需求传递中的信息损耗

| 环节 | 问题 | 结果 |
|------|------|------|
| 业务方→产品经理 | 口头描述模糊 | "我想要一个看板" |
| 产品经理→设计师 | 理解偏差 | 设计稿不是业务方想要的 |
| 设计师→开发者 | 交互细节缺失 | 开发成果与预期不符 |
| 开发→上线 | 技术约束未提前暴露 | 返工或妥协 |

### 2.3 现有工具的错位

```
现有工具定位：

v0.dev ─────────→ 生成生产代码（开发者导向）
                    ↓
Figma AI ───────→ 生成设计稿（设计师导向）
                    ↓
Coze/n8n ───────→ 工作流自动化（执行导向）

缺失：专门的"需求验证"层（业务人员导向）
```

## 3. 业务场景

### 3.1 场景一：快速需求验证

**参与者**: 产品经理
**前提条件**: 有新功能想法，需要向团队展示

**主流程**:
1. PM 用自然语言描述需求："我想要一个客户跟进系统，能按状态分组"
2. MockCraft 生成可交互 Demo
3. PM 体验 Demo，提供反馈："需要加上筛选功能"
4. MockCraft 更新 Demo
5. PM 确认 Demo 符合预期
6. 导出 PRD 和原型给研发团队

**预期结果**: 5分钟内完成需求澄清，获得可分享的原型

### 3.2 场景二：需求评审辅助

**参与者**: 业务方 + 研发团队
**前提条件**: 需求评审会议，需要统一理解

**主流程**:
1. 业务方在 MockCraft 中描述需求
2. 生成 Demo 投影展示
3. 研发团队提问，业务方实时调整描述
4. 迭代至双方达成一致
5. 自动生成决策历史文档

**预期结果**: 评审会上即时澄清分歧，减少会后返工

### 3.3 场景三：复杂场景判断

**参与者**: 产品经理
**前提条件**: 不确定需求该用什么工具实现

**主流程**:
1. PM 描述需求
2. MockCraft 分析需求复杂度
3. 给出建议："此需求可用飞书多维表格实现" 或 "建议用 DDDW 开发"
4. 生成对应的配置/PRD

**预期结果**: 快速选择合适的实现路径

## 4. 功能需求

### 4.1 核心功能

#### FR-001: 自然语言意图输入
- **描述**: 用户通过自然语言描述需求
- **优先级**: P0
- **验收标准**:
  - 支持中文输入
  - 最少 5 个字即可开始
  - 支持多轮对话式补充

#### FR-002: 意图提取与结构化
- **描述**: 系统从自然语言中提取关键信息并结构化
- **优先级**: P0
- **验收标准**:
  - 提取实体类型（如看板、表单、列表）
  - 提取功能需求（如筛选、分组、排序）
  - 提取数据字段

#### FR-003: 可交互原型生成
- **描述**: 根据结构化意图生成可交互的 HTML 原型
- **优先级**: P0
- **验收标准**:
  - 生成中保真原型（非线框，但也不精致）
  - 支持基本交互（点击、输入、筛选）
  - 响应式布局，支持移动端预览

#### FR-004: 多场景覆盖
- **描述**: 自动生成边界场景的模拟
- **优先级**: P1
- **验收标准**:
  - 空状态显示
  - 大量数据状态
  - 错误状态

#### FR-005: 渐进式迭代
- **描述**: 支持基于反馈快速修改原型
- **优先级**: P0
- **验收标准**:
  - 用户可通过自然语言描述修改
  - 修改后秒级重新生成
  - 保留历史版本对比

#### FR-006: PRD 自动生成
- **描述**: 根据迭代历史生成需求文档
- **优先级**: P1
- **验收标准**:
  - 包含需求演进记录
  - 包含最终确认的功能点
  - Markdown 格式，便于版本控制

#### FR-007: 导出到下游工具
- **描述**: 支持将确认后的需求导出到各种工具
- **优先级**: P1
- **验收标准**:
  - 导出 Markdown PRD
  - 导出 HTML 原型
  - 导出飞书多维表格配置（JSON）

### 4.2 辅助功能

#### FR-008: 需求复杂度分析
- **描述**: 分析需求复杂度并给出实现建议
- **优先级**: P2
- **验收标准**:
  - 标记简单/中等/复杂
  - 推荐合适的实现工具

#### FR-009: 协作分享
- **描述**: 支持分享原型链接给他人查看
- **优先级**: P2
- **验收标准**:
  - 生成分享链接
  - 支持查看模式（不可编辑）

## 5. 非功能需求

### 5.1 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 原型生成时间 | < 10s | 从输入到可查看原型 |
| 修改迭代时间 | < 3s | 基于反馈的重新生成 |
| 页面加载时间 | < 2s | 首次访问工具 |

### 5.2 可用性需求

- **门槛**: 无需学习成本，业务人员可直接使用
- **语言**: 中文界面优先
- **反馈**: 每个操作都有明确的系统反馈

### 5.3 可维护性需求

- **代码**: 开源，易于社区贡献
- **文档**: 完整的开发文档和 API 文档
- **测试**: 核心功能有单元测试覆盖

### 5.4 安全需求

- **隐私**: 用户输入数据不永久存储（或本地存储）
- **LLM**: 支持配置自己的 API Key

## 6. 约束条件

### 6.1 技术约束

- **前端**: 纯前端实现，可部署为静态页面
- **LLM**: 优先使用 OpenAI/Claude API，支持自定义
- **输出**: 原型使用 HTML/CSS/JS，PRD 使用 Markdown

### 6.2 产品约束

- **定位**: 不做生产代码生成，专注需求澄清
- **竞争**: 不与 v0.dev/Figma 直接竞争，形成互补
- **形态**: 开源工具而非商业 SaaS

### 6.3 时间约束

- **MVP**: 2-4 周内可演示核心流程
- **完整版**: 3 个月内开源发布

## 7. 验收标准

### 7.1 核心流程验收

```gherkin
场景: 完整的需求澄清流程
  假设 用户打开 MockCraft
  当 用户输入 "我想要一个客户跟进系统，能按状态分组"
  那么 系统应该在 10 秒内生成可交互原型
  当 用户查看原型并提供反馈 "需要加上筛选功能"
  那么 系统应该在 3 秒内更新原型
  当 用户确认需求
  那么 系统应该生成包含决策历史的 PRD
  并且 支持导出为 Markdown 和 HTML
```

### 7.2 质量验收清单

- [ ] 业务人员无需培训即可使用
- [ ] 原型修改成本趋近于零（心理层面）
- [ ] 生成的 PRD 开发者能理解并参考
- [ ] 与现有工具（飞书/Coze/Figma）形成良好衔接
