# 内部效率平台设计方案

> 面向非技术人员的工具生成平台

---

## 一、平台定位

### 1.1 目标

为组织内部非技术人员提供自动化工具生成能力，通过自然语言交互快速创建个人效率工具。

### 1.2 用户群体

- 产品、运营、销售等业务人员
- 无编程背景但有自动化诉求的员工
- 需要快速原型验证的团队

### 1.3 核心能力

| 能力 | 说明 |
|------|------|
| 自然语言输入 | 用户用日常语言描述需求 |
| 现有工具推荐 | 优先推荐内部已有系统或标准工具 |
| 可视化确认 | 通过 Mock 展示确认需求理解 |
| 代码生成 | 输出可运行的定制化工具 |
| 一键部署 | 平台托管运行，无需用户配置 |

## 三、核心流程设计

### 3.1 流程概览

```
用户输入需求
    ↓
┌─────────────────────────────────────────┐
│ Step 0: 内部系统检查                     │
│ ─────────────────────                   │
│ • 检索内部已有系统                        │
│ • 检索标准工具库                          │
│ • 判断是否可直接使用                      │
│                                         │
│ 结果:                                    │
│ ✓ 有现有系统 → 引导使用 (流程结束)        │
│ ✗ 无合适系统 → 进入 Step 1               │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Step 1: 需求澄清                         │
│ ─────────────                           │
│ • 多轮对话确认细节                        │
│ • 提取功能点                              │
│ • 识别数据流                              │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Step 2: Mock 展示                        │
│ ─────────────                           │
│ • 生成界面描述                           │
│ • 展示交互流程                           │
│ • 用户确认或修改                         │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Step 3: 代码生成                         │
│ ─────────────                           │
│ • 选择技术模板                           │
│ • 组装业务代码                           │
│ • 安全审查                               │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Step 4: 部署运行                         │
│ ─────────────                           │
│ • 自动构建镜像                           │
│ • 分配运行环境                           │
│ • 提供访问入口                           │
└─────────────────────────────────────────┘
    ↓
用户获得可运行的工具
```

### 3.2 各阶段详细设计

#### Step 0: 内部系统检查

**检查内容**：
1. 内部已有系统清单（OA、CRM、ERP 等）
2. 标准工具库（审批、表单、报表等）
3. 低代码平台能力

**输出示例**：
```
用户: "我想做一个请假统计表"

系统: "发现以下现有方案:
  1. OA 系统 - 已有请假模块，支持导出统计
  2. 内部报表平台 - 可配置化生成统计视图
  
  建议使用现有方案，无需开发新工具。
  如需特殊功能，继续创建定制工具。"
```

#### Step 1: 需求澄清

**对话示例**：
```
用户: "我想要一个客户跟进工具"

系统: "请描述具体需求:
  1. 需要记录哪些信息? (客户名、联系方式、跟进状态...)
  2. 几个人使用? (个人/团队)
  3. 需要提醒功能吗? (跟进提醒)
  4. 需要导出报表吗?"

用户: "记录客户名、电话、上次联系时间，需要提醒3天未联系的客户"

系统: "理解，这是一个客户跟进提醒工具，主要功能:
  - 客户信息管理
  - 联系记录
  - 3天未联系自动提醒
  确认吗?"
```

#### Step 2: Mock 展示

**展示形式**：
- 界面布局描述
- 简单的线框图
- 交互流程说明

**示例**：
```
【客户跟进工具 - 界面设计】

┌─────────────────────────────────┐
│  客户跟进表                      │
├─────────────────────────────────┤
│ [添加客户] [导出数据]           │
├─────────────────────────────────┤
│ 客户名    电话      状态    操作 │
│ ─────────────────────────────────│
│ 张三      138xxx   待跟进  [编辑]│
│ 李四      139xxx   已联系  [编辑]│
│ 王五      137xxx   ⚠️3天未联系  │
└─────────────────────────────────┘

提醒功能: 每天9点检查，3天未联系的客户标红提醒
```

#### Step 3: 代码生成

**技术栈选择**：
- 前端：React/Vue + Ant Design（内部统一组件库）
- 后端：Node.js/Python（根据场景）
- 存储：内部数据库服务
- 部署：容器化，平台托管

**生成规范**：
- 使用内部标准模板
- 集成统一登录
- 符合安全规范
- 可维护的代码结构

#### Step 4: 部署运行

**自动化部署**：
- CI/CD 流水线自动构建
- 自动分配测试环境
- 一键发布到生产
- 自动配置域名/访问权限

**交付物**：
- 工具访问链接
- 使用说明文档
- 管理员后台

## 四、技术实现方案

### 4.1 核心服务模块

#### NLU 服务

**功能**：
- 意图识别（创建工具/查询工具/修改工具）
- 实体提取（功能点、数据字段、业务规则）
- 需求分类（表单类/流程类/报表类）

**实现**：
- 基于大模型 API
- 内部领域知识库增强
- 多轮对话管理

#### Mock 生成服务

**功能**：
- 界面布局生成
- 交互流程设计
- 数据模型定义

**输出**：
- 结构化描述
- 简单可视化预览
- 用户确认接口

#### 代码生成服务

**功能**：
- 模板匹配
- 代码组装
- 安全审查

**模板库**：
- 表单类工具模板
- 流程类工具模板
- 报表类工具模板
- 自定义扩展点

### 4.2 基础设施

#### 运行环境

- 基于 Kubernetes 的容器平台
- 每个工具独立命名空间
- 资源配额管理
- 自动扩缩容

#### 数据存储

- 应用数据：内部数据库服务（MySQL/PostgreSQL）
- 文件存储：内部对象存储
- 缓存：Redis 集群

#### 安全体系

- 统一身份认证（SSO）
- 权限控制（RBAC）
- 数据隔离
- 审计日志

### 4.3 集成接口

#### 内部系统集成

- OA 系统 API
- HR 系统 API
- 财务系统 API
- 统一消息服务

#### 标准组件库

- 内部 UI 组件库
- 标准表单组件
- 通用图表组件
- 审批流程组件

## 六、使用流程示例

### 场景：创建一个客户拜访记录工具

```
【用户】产品经理小李 (非技术背景)

Step 0: 进入平台
- 打开内部效率平台
- SSO 自动登录

Step 1: 描述需求
小李: "我需要记录客户拜访情况，
       包括拜访时间、客户反馈、下一步计划，
       最好能提醒我该回访的客户"

Step 2: 系统检查
系统: "检索内部系统...
       未发现完全匹配的系统。
       建议创建定制工具。"

Step 3: 需求澄清
系统: "确认几个细节:
       1. 只是你自己用还是团队共享?
       2. 几天未回访需要提醒?
       3. 需要导出报表吗?"

小李: "团队使用，7天未回访提醒，需要月度报表"

Step 4: Mock 展示
系统: "这是设计的界面..."
       [展示界面描述]

小李: "可以，但希望加上客户行业分类"

Step 5: 调整确认
系统: "已调整，增加行业字段..."

Step 6: 代码生成
系统: "正在生成代码..."
       [后台自动构建]

Step 7: 部署交付
系统: "工具已部署: https://tools.company.com/crm-x7k9j"
       "使用说明已发送至企业微信"

【结果】
- 小李获得定制化工具
- 团队成员可访问使用
- 数据存储在公司内部
- 平台统一运维管理
```

## 七、项目边界

### 7.1 平台负责

- 需求理解到代码生成的全流程
- 工具的运行环境托管
- 数据的安全存储
- 统一认证和权限
- 基础运维监控

### 7.2 平台不负责

- 复杂的业务逻辑定制（需人工介入）
- 与外部系统的对接（评估后支持）
- 大规模数据处理（有独立系统）
- 核心生产系统（走正式开发流程）

## 八、演进路线

### Phase 1: MVP (3个月)
- 基础流程跑通
- 支持表单类工具
- Web 界面为主
- 内部小范围试用

### Phase 2: 完善 (3个月)
- 增加流程类工具
- 支持企业微信/钉钉
- 模板库丰富
- 全公司推广

### Phase 3: 智能化 (6个月)
- 需求预测
- 智能推荐
- 自动化测试
- 生态集成
