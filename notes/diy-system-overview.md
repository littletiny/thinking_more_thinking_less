# DIY 开发系统总览

> 面向非技术人员的自动化工具生成系统

---

## 一、系统定位

### 1.1 一句话定义

通过自然语言交互，为每个人生成定制化自动化工具的脚手架系统。

### 1.2 核心区别

| 维度 | 低代码平台 | 传统开发 | DIY 系统 |
|------|-----------|---------|---------|
| 用户 | 技术人员 | 程序员 | **普通人** |
| 门槛 | 需学习平台 | 需学编程 | **自然语言** |
| 产出 | 平台依赖 | 代码 | **独立运行的工具** |
| 成本 | 订阅费 | 人工费 | **免费 + 自托管** |

### 1.3 适用场景

**推荐 DIY**：
- 极度个性化需求（如"记录和谁吃饭"）
- 隐私敏感（数据不能上云）
- 小众场景（现有工具不满足）

**推荐现有工具**：
- 简单记账 → 随手记
- 待办提醒 → 滴答清单
- 笔记整理 → Notion

## 二、核心理念（3 个支柱）

### 支柱 1：解决问题优先（Solution First）

> 不是给 coding 能力，是找到最优解决方案

**关键流程**：
```
用户提出需求
    ↓
Step 0: 现有方案检查（新增）
  - WebSearch 搜索现有工具
  - 向用户展示 2-3 个选项
  - 如果满足，推荐现有工具（结束）
  - 如果不满足，继续 DIY
    ↓
DIY 流程
```

**原则**：能简单解决就不复杂解决

### 支柱 2：看见的能力（Power of Seeing）

> 人无法想象自己不知道的东西，必须看见才能判断

**实现方式**：
- 用 **Mock** 展示可能性
- 文字描述界面 + ASCII 图表
- 用户基于看见反馈

**关键洞察**：展示先于询问

### 支柱 3：脚手架而非平台（Scaffold, Not Platform）

> 给工具，不给服务；用户拥有，平台不管

**核心特征**：
- 生成源代码即结束
- 用户自托管（本地/云/树莓派）
- 开发者零运维成本
- 独立运行，不依赖平台

## 三、系统架构（3 层）

```
┌─────────────────────────────────────────┐
│  接入层（多种形式）                      │
│  CLI / IM Bot / Web Demo                │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  核心引擎层（独立，0依赖）               │
│  NLU → Mock → Code Generator            │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  流程编排层（5 步骤）                    │
│  需求检查 → 澄清 → Mock → 生成 → 交付   │
└─────────────────────────────────────────┘
```

### 3.1 核心引擎

| 模块 | 功能 | 技术 |
|------|------|------|
| **NLU** | 理解自然语言需求 | 大模型 API |
| **Mock** | 生成可视化原型 | 模板 + 描述 |
| **Generator** | 输出可运行代码 | 代码模板库 |

### 3.2 技术规范（反工程）

**13 条 DIY 规范总结**：

| 规范 | 核心 | 反模式 |
|------|------|--------|
| 存储 | JSON 文件 | 数据库/ORM |
| 前端 | 单文件内联 | 组件化/构建 |
| 架构 | 单文件平铺 | 分层/MVC |
| 代码 | 简单直接 | 严格规范 |
| 安全 | **唯一严格** | - |
| 测试 | 手动即可 | 自动化测试 |
| 依赖 | 复制粘贴 | npm install |

**口诀**：能用就行，安全除外

## 四、完整流程（5 步骤）

### Step 0: 现有方案检查（关键新增）

**目的**：避免重复造轮子

**执行**：
1. WebSearch 搜索相关工具
2. 向用户展示 2-3 个现有选项
3. 用户选择：使用现有 / 继续 DIY

### Step 1: 需求澄清

- 自然语言多轮对话
- 提取功能点、数据字段、业务规则

### Step 2: Mock 演示

- 文字描述界面
- ASCII 图表展示
- 用户确认或修改

### Step 3: 代码生成

- 模板匹配
- 代码组装
- 安全审查

### Step 4: 交付

- 源代码压缩包
- README + 运行脚本
- 用户自托管

## 七、关键决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| 平台 vs 脚手架 | **脚手架** | 用户拥有，零运维 |
| GUI vs 自然语言 | **自然语言** | 成本可接受，体验好 |
| 托管 vs 自托管 | **自托管** | 零运维成本，数据自主 |
| 商业 vs 普惠 | **普惠** | 解决问题而非赚钱 |
