# DIY Scaffold 完整项目架构

> 从需求到代码的自动化脚手架系统

---

## 一、系统概述

### 1.1 定位

为个人用户提供的自动化工具生成脚手架，通过自然语言交互生成可运行的定制化工具代码。

### 1.2 核心特性

- **自然语言交互**：用户通过对话描述需求
- **现有方案检查**：优先推荐现有工具，避免重复造轮子
- **Mock 驱动验证**：通过 Demo 展示确认需求
- **代码生成**：输出可直接运行的源代码
- **多形态交付**：CLI / IM Bot / Web Demo

## 三、核心流程详解

### 3.1 Step 0: 现有方案验证

**目的**：避免重复造轮子，优先使用现有工具

**执行**：
1. WebSearch 搜索相关工具
2. 模型知识库检索常见方案
3. 向用户展示 2-3 个现有选项
4. 用户选择：使用现有工具 / 继续 DIY

**示例**：
```
用户: "我想做记账工具"

系统: "发现以下现有方案：
  1. 随手记 - 免费，支持自动分类
  2. MoneyWiz - 付费，多账户管理
  3. Firefly III - 开源，自托管
  
  这些能满足你的需求吗？"

用户: "我需要记录'和谁吃饭'，这些都不行"
→ 进入 DIY 流程
```

### 3.2 Step 1: 需求确认

**目的**：通过对话澄清具体需求

**方式**：自然语言多轮对话

**输出**：结构化的需求文档

### 3.3 Step 2: Mock 演示

**目的**：让用户"看见"最终效果

**形式**：
- 文字描述界面布局
- ASCII 图表展示
- 简单的交互流程说明

**迭代**：用户反馈 → 调整 Mock → 再次确认

### 3.4 Step 3: 代码生成

**输入**：确认的需求 + Mock 设计

**输出**：
- 完整源代码
- 运行说明 README
- 依赖清单

**技术规范**：
- 存储：JSON 文件（非数据库）
- 架构：单文件优先（非分层）
- 代码风格：简单直接
- 安全：输入验证、路径限制

### 3.5 Step 4: 交付

**交付物**：
- 源代码压缩包
- 部署文档
- 运行脚本

**部署方式**：
- 本地运行（Node.js/Python）
- 云服务器部署
- 打包为 Desktop App

## 四、技术规范 (DIY-001~013)

### 4.1 存储规范 (DIY-001)

- 使用 JSON 文件存储数据
- 每个功能独立数据文件
- 全量读写（数据量 < 10MB）

### 4.2 前端规范 (DIY-002)

- 单 HTML 文件包含所有代码
- 内联样式和脚本
- 使用 CDN 引入第三方库

### 4.3 架构规范 (DIY-003)

- 单文件实现完整功能
- 函数即接口
- 平铺代码结构

### 4.4 代码风格 (DIY-004)

- 无严格规范
- console.log 调试
- 变量名简洁

### 4.5 安全规范 (DIY-005)

- 输入验证（防注入）
- 文件路径限制
- 敏感信息保护

### 4.6 测试规范 (DIY-006)

- 手动测试
- 无需自动化测试框架

### 4.7 依赖规范 (DIY-007)

- 优先使用标准库
- 复制粘贴小函数
- 延迟添加依赖

### 4.8 现有方案检查 (DIY-008)

- 必须搜索现有工具
- 向用户展示选项
- 优先推荐现有方案

### 4.9 Mock 驱动 (DIY-009)

- 展示先于询问
- 提供具体示例
- 用户基于看见判断

### 4.10 市场认知 (DIY-010)

- 了解现有工具能力
- 明确 DIY 适用范围

### 4.11 成本模式 (DIY-011)

- 一锤子买卖
- 用户自托管
- 开发者零运维

### 4.12 错误处理 (DIY-012)

- 简单错误处理
- 崩溃即调试信息

### 4.13 解决问题优先 (DIY-013)

- 不是给 coding 能力
- 是找到最优解决方案
- 能简单就不复杂

## 七、适用场景

### 7.1 推荐 DIY

- 极度个性化需求
- 隐私敏感（数据不能上云）
- 小众场景
- 现有工具完全不满足

### 7.2 推荐现有工具

- 简单记账 → 随手记
- 待办提醒 → 滴答清单
- 笔记整理 → Notion
- 自动化 → Zapier
